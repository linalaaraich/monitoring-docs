Sprint 2 Extension Overview

Sprint 2 was originally scheduled to end April 16, but was extended to April 23 to accommodate two new epics that significantly change the platform architecture. This document covers every piece of work added during the extension, in enough detail to replicate from scratch.

What Changed

  • Epic 3: K8s Migration — 10 stories, 34 story points. Migrates monitored infrastructure (Spring Boot, Kong, AI pipeline) from per-VM docker-compose to a single-node k3s cluster.
  • Epic 4: AI Hardening — 3 stories, 8 story points. Adds circuit breaker, JSON schema enforcement, and self-observability metrics to the AI RCA pipeline.
  • 3 existing stories rescoped: SCRUM-63 (Terraform) restructured from 4 EC2s to 3; SCRUM-47 (Prometheus targets) gains kubernetes_sd_configs; SCRUM-51 (alert rules) gains Jinja2 conditional selectors.
  • 2 existing stories superseded: E1-05 (VM-target E2E validation) replaced by E3-10 (K8s E2E validation); original per-VM OTel Collector story subsumed by E3-07 (DaemonSet).

Terraform Rescope (SCRUM-63)

What Changed

The original Terraform provisioned 4 separate EC2 instances (monitoring, backend, network, ai). The rescoped version provisions 3 instances:

  • 1 monitoring EC2 — unchanged, t3.large. Runs Prometheus, Grafana, Loki, Jaeger.
  • 1 k3s EC2 — new, t3.xlarge (4 vCPU, 16GB RAM). Runs the single-node k3s cluster with Spring Boot, Kong, AI stack, OTel Collector.
  • 1 GPU EC2 — renamed from "ai", g4dn.xlarge. For Ollama if a separate GPU node is needed (optional — can colocate on k3s node if it has a GPU).

AMI Pinning Explanation

By default, the ubuntu_ami_id variable is empty, which causes Terraform to look up the latest Ubuntu 22.04 LTS AMI via a data.aws_ami data source. This is convenient for development but risky for production — a new AMI might introduce breaking changes.

To pin a specific AMI (recommended for reproducible builds):

# In terraform.tfvars:
ubuntu_ami_id = "ami-0abcdef1234567890"

# This skips the data source lookup and uses the exact AMI.
# Find your AMI with:
aws ec2 describe-images \
  --owners 099720109477 \
  --filters "Name=name,Values=ubuntu/images/hvm-ssd/ubuntu-jammy-22.04-amd64-server-*" \
  --query 'sort_by(Images, &CreationDate)[-1].ImageId' \
  --output text

k3s Bootstrap (E3-01)

What is xanmanning.k3s?

The xanmanning.k3s Ansible role (version 3.4.4) is a community-maintained role that fully automates the k3s lifecycle. It handles:

  • Installation: Downloads the k3s binary, creates the systemd service, starts k3s
  • Configuration: Applies server flags (disable Traefik, set kubeconfig permissions, etc.)
  • Multi-node: Supports adding agent nodes to a server (not used yet, but ready for scale-out)
  • Upgrade: Can pin a specific k3s version and upgrade in place
  • Uninstall: Cleans up k3s completely (k3s-uninstall.sh)

How requirements.yml Was Updated

# requirements.yml (add this entry under the roles section)
roles:
  # ... existing roles ...
  - name: xanmanning.k3s
    version: "v3.4.4"

# Install the role:
ansible-galaxy install -r requirements.yml --force
Note: The --force flag re-downloads even if the role exists. Use it when upgrading versions. Without it, an already-installed role is skipped.

The k3s.yml Playbook Step by Step

# playbooks/k3s.yml
---
- name: Bootstrap k3s cluster
  hosts: k3s_cluster
  become: true
  roles:
    - role: xanmanning.k3s      # Installs and configures k3s

  post_tasks:
    # Fetch the kubeconfig to the Ansible controller
    - name: Fetch kubeconfig from k3s node
      fetch:
        src: /etc/rancher/k3s/k3s.yaml
        dest: "{{ playbook_dir }}/../kubeconfig"
        flat: true

    # Rewrite the kubeconfig to use the node's real IP (not 127.0.0.1)
    - name: Update kubeconfig server address
      delegate_to: localhost
      become: false
      replace:
        path: "{{ playbook_dir }}/../kubeconfig"
        regexp: 'https://127\.0\.0\.1:6443'
        replace: "https://{{ ansible_host }}:6443"

    # Install Helm on the k3s node
    - name: Install Helm
      shell: |
        curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash
      args:
        creates: /usr/local/bin/helm

What the Kubeconfig Fetch Does

k3s writes its kubeconfig to /etc/rancher/k3s/k3s.yaml on the server node. This file contains the cluster CA certificate, a client certificate, and points to https://127.0.0.1:6443. The playbook:

  1. Fetches the file to the Ansible controller at ./kubeconfig
  2. Rewrites the server address from 127.0.0.1 to the node's actual IP so you can use kubectl remotely
  3. You can then do: export KUBECONFIG=$(pwd)/kubeconfig and run kubectl from your local machine

How to Verify

# On the k3s node directly:
sudo k3s kubectl get nodes

# Expected output:
# NAME       STATUS   ROLES                  AGE   VERSION
# k3s-node   Ready    control-plane,master   1m    v1.30.6+k3s1

# From your local machine (after kubeconfig fetch):
export KUBECONFIG=$(pwd)/kubeconfig
kubectl get nodes
kubectl get pods -A       # shows kube-system pods (coredns, local-path-provisioner, metrics-server)

Container Images (E3-02)

Why We Do Not Use a Registry

In a typical Kubernetes setup, container images live in a registry (Docker Hub, ECR, GCR). Pods pull images from the registry at deploy time. However, for our single-node k3s cluster with 1-2 contributors, a registry adds unnecessary complexity:

  • ECR: Requires IAM credentials, push/pull policies, lifecycle rules, and costs money
  • Self-hosted registry: Another service to maintain, secure, and back up
  • Docker Hub: Rate limits on pulls, public exposure of images

Instead, we build images locally with Docker, save them as tarballs, and import them directly into k3s's containerd. This is fast, simple, and works perfectly for single-node.

The k3s_images Ansible Role

The k3s_images Ansible role automates the entire image pipeline. It is structured as follows:

roles/k3s_images/
  tasks/
    main.yml          # Orchestrates the build + import flow
  defaults/
    main.yml          # Default image names and tags
  files/              # (empty, source repos synced at runtime)

How synchronize Sends Source Repos to the k3s Host

Each first-party service (triage-service, mcp-prometheus, etc.) lives in a source repository on the Ansible controller. The role uses Ansible's synchronize module (which wraps rsync) to push the source code to the k3s node:

# Inside roles/k3s_images/tasks/main.yml
- name: Sync source code for {{ item.name }}
  synchronize:
    src: "{{ item.src_path }}/"
    dest: "/tmp/build/{{ item.name }}/"
    delete: true
    rsync_opts:
      - "--exclude=.git"
      - "--exclude=node_modules"
      - "--exclude=__pycache__"
  loop: "{{ first_party_images }}"

# first_party_images is defined in defaults/main.yml:
# first_party_images:
#   - name: triage-service
#     src_path: "{{ playbook_dir }}/../services/triage-service"
#     image: cires/triage-service
#   - name: mcp-prometheus
#     src_path: "{{ playbook_dir }}/../services/mcp-prometheus"
#     image: cires/mcp-prometheus
#   ... (one entry per first-party image)

The Build Flow: docker build, docker save, k3s ctr images import

After syncing the source, the role executes three steps per image:

# Step 1: Build the Docker image on the k3s host
# (Docker must be installed alongside k3s for building)
- name: Build image {{ item.image }}:{{ image_tag }}
  community.docker.docker_image:
    name: "{{ item.image }}"
    tag: "{{ image_tag }}"
    source: build
    build:
      path: "/tmp/build/{{ item.name }}"
    force_source: true
  loop: "{{ first_party_images }}"

# Step 2: Save the image to a tarball
- name: Save image to tarball
  command: >
    docker save {{ item.image }}:{{ image_tag }}
    -o /tmp/{{ item.name }}.tar
  loop: "{{ first_party_images }}"

# Step 3: Import into k3s containerd
- name: Import image into k3s
  command: >
    k3s ctr images import /tmp/{{ item.name }}.tar
  loop: "{{ first_party_images }}"

# Step 4: Cleanup tarballs
- name: Remove tarballs
  file:
    path: "/tmp/{{ item.name }}.tar"
    state: absent
  loop: "{{ first_party_images }}"
Why Docker + k3s ctr? k3s uses containerd, which cannot build images. Docker can build images but k3s does not use Docker's image store. So we build with Docker, export to a tarball, and import into containerd. This is the standard pattern for air-gapped or registryless k3s deployments.

Critical: Helm charts must set imagePullPolicy: IfNotPresent (not Always). If set to Always, k3s will try to pull from a registry and fail with ImagePullBackOff because these images do not exist in any registry.

Upgrade Path to ECR (Multi-Node)

When the project moves to multi-node k3s or EKS, the image strategy must change because images imported on one node are not visible to other nodes. The upgrade path:

  1. Create an ECR repository per image (aws ecr create-repository --repository-name cires/triage-service)
  2. Authenticate Docker to ECR: aws ecr get-login-password | docker login --username AWS --password-stdin ECR_URL
  3. Push images: docker tag cires/triage-service:latest ECR_URL/cires/triage-service:latest && docker push ...
  4. Update Helm charts to use the ECR URL as the image repository
  5. Add an ECR pull secret to each namespace (kubectl create secret docker-registry ...)
  6. Remove the k3s_images role from the playbook (no longer needed)

Helm Charts Deep Dive (E3-03, E3-04, E3-05)

Every Helm chart follows the same directory layout:

charts/<chart-name>/
  Chart.yaml          # Chart metadata: name, version, appVersion, description

  templates/          # Kubernetes manifest templates using Go template syntax

_helpers.tpl Explained

The _helpers.tpl file defines named templates that are reused across all manifests. This avoids repetition:

{{/* Generate the full name: release-name + chart-name, truncated to 63 chars */}}
{{- define "spring-boot.fullname" -}}
{{- printf "%s-%s" .Release.Name .Chart.Name | trunc 63 | trimSuffix "-" }}
{{- end }}

{{/* Common labels applied to every resource */}}
{{- define "spring-boot.labels" -}}
helm.sh/chart: {{ .Chart.Name }}-{{ .Chart.Version }}
app.kubernetes.io/name: {{ .Chart.Name }}

{{- end }}

{{/* Selector labels (used in matchLabels and service selectors) */}}

{{- end }}

OTel Collector DaemonSet (E3-07)

What is a DaemonSet and Why?

A DaemonSet is a Kubernetes resource that ensures one pod runs on every node in the cluster. When a new node is added, a DaemonSet pod is automatically scheduled on it. When a node is removed, the pod is garbage-collected.

We use a DaemonSet for the OTel Collector because:

  • It needs access to /var/log/pods/ on each node to read container logs
  • It needs to run on every node to collect logs and receive traces from all pods
  • If we add more k3s nodes later, the DaemonSet automatically scales with the cluster

The OTel Collector Config Explained

The Collector is configured via a ConfigMap. It has three sections: receivers (how data enters), processors (how data is enriched), and exporters (where data goes).

# manifests/otel-collector/configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: otel-collector-config
  namespace: observability
data:
  otel-collector-config.yaml: |
    receivers:
      # === FILELOG RECEIVER ===
      # Reads container logs from the node filesystem
      filelog:
        include:
          - /var/log/pods/*/*/*.log
        start_at: beginning
        include_file_path: true
        include_file_name: false
        operators:
          # Parse the container runtime log format (containerd uses CRI format)
          - type: regex_parser
            regex: '^(?P<time>[^ ]+) (?P<stream>stdout|stderr) (?P<logtag>[^ ]*) ?(?P<log>.*)$'
            timestamp:
              parse_from: attributes.time
              layout: '%Y-%m-%dT%H:%M:%S.%LZ'
          # Extract K8s metadata from the file path
          # Path format: /var/log/pods/NAMESPACE_PODNAME_UID/CONTAINER/0.log
          - type: regex_parser
            regex: '^.*\/(?P<namespace>[^_]+)_(?P<pod>[^_]+)_(?P<uid>[^\/]+)\/(?P<container>[^\/]+)\/'
            parse_from: attributes["log.file.path"]

      # === OTLP RECEIVER ===
      # Receives traces and metrics from application pods via gRPC and HTTP
      otlp:
        protocols:
          grpc:
            endpoint: 0.0.0.0:4317
          http:
            endpoint: 0.0.0.0:4318

    processors:
      # === K8S ATTRIBUTES PROCESSOR ===
      # Enriches telemetry with K8s metadata (pod name, namespace, node, labels)
      k8sattributes:
        auth_type: "serviceAccount"
        passthrough: false
        extract:
          metadata:
            - k8s.pod.name
            - k8s.pod.uid
            - k8s.namespace.name
            - k8s.node.name
            - k8s.deployment.name
          labels:
            - tag_name: app
              key: app.kubernetes.io/name
              from: pod
        pod_association:
          - sources:
              - from: resource_attribute
                name: k8s.pod.ip

      # === BATCH PROCESSOR ===
      # Batches telemetry for efficient export
      batch:
        timeout: 5s
        send_batch_size: 1024

    exporters:
      # === LOKI EXPORTER ===
      # Sends logs to Loki on the monitoring VM
      loki:
        endpoint: "http://{{ monitoring_ip }}:3100/loki/api/v1/push"
        labels:
          attributes:
            k8s.namespace.name: "namespace"
            k8s.pod.name: "pod"
            k8s.container.name: "container"
            app: "app"

      # === OTLP EXPORTER ===
      # Sends traces to OTel Collector on the monitoring VM, which forwards to Jaeger
      otlp:
        endpoint: "http://{{ monitoring_ip }}:4317"
        tls:
          insecure: true

    service:
      pipelines:
        logs:
          receivers: [filelog]
          processors: [k8sattributes, batch]
          exporters: [loki]
        traces:
          receivers: [otlp]
          processors: [k8sattributes, batch]
          exporters: [otlp]

Prometheus K8s Discovery (E3-08)

What is kubernetes_sd_configs?

kubernetes_sd_configs is a Prometheus service discovery mechanism that automatically finds scrape targets by querying the Kubernetes API. Instead of maintaining a static list of IPs and ports in prometheus.yml, Prometheus asks K8s: "give me all pods that have the annotation prometheus.io/scrape: true."

Why It Replaces Static Targets

  • Pods are ephemeral: A pod's IP changes every time it restarts. Static targets would break constantly.
  • Auto-discovery: Deploy a new service with the right annotations and Prometheus picks it up automatically — no config change needed.
  • Rich metadata: K8s SD provides labels like pod name, namespace, node name, and all pod labels/annotations, which become Prometheus labels for filtering and grouping.

The RBAC Needed

Prometheus runs on the monitoring VM (outside the cluster), so it authenticates to the K8s API using a ServiceAccount token. This requires RBAC resources inside the k3s cluster:

kind: Namespace
metadata:
  name: observability
---

  name: prometheus-scraper
  namespace: observability
---

  name: prometheus-scraper
rules:
  - apiGroups: [""]
    resources: ["nodes", "nodes/proxy", "services", "endpoints", "pods"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["extensions", "networking.k8s.io"]
    resources: ["ingresses"]
    verbs: ["get", "list", "watch"]

  - job_name: 'kubernetes-pods'
    kubernetes_sd_configs:
      - api_server: 'https://{{ k3s_api_server }}:6443'

          insecure_skip_verify: true    # k3s uses self-signed certs

    relabel_configs:

      # 1. Only scrape pods that have prometheus.io/scrape = "true"
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_scrape]
        action: keep
        regex: true

      # 2. Override the metrics path if the pod specifies one
      #    e.g., prometheus.io/path: "/actuator/prometheus"
      - source_labels: [__meta_kubernetes_pod_annotation_prometheus_io_path]
        action: replace
        target_label: __metrics_path__
        regex: (.+)

      # 3. Override the port if the pod specifies one
      #    e.g., prometheus.io/port: "8080"
      #    Constructs the target address as pod_ip:custom_port
      - source_labels: [__address__, __meta_kubernetes_pod_annotation_prometheus_io_port]
        action: replace
        regex: ([^:]+)(?::\d+)?;(\d+)
        replacement: $1:$2
        target_label: __address__

      # 4. Add the pod's namespace as a "namespace" label
      - source_labels: [__meta_kubernetes_namespace]
        action: replace
        target_label: namespace

      # 5. Add the pod name as a "pod" label
      - source_labels: [__meta_kubernetes_pod_name]
        action: replace
        target_label: pod

      # 6. Add the app.kubernetes.io/name label as "app"
      #    This is set by our Helm charts and is the primary selector for alerts
      - source_labels: [__meta_kubernetes_pod_label_app_kubernetes_io_name]
        action: replace
        target_label: app

      # 7. Keep the node name for troubleshooting
      - source_labels: [__meta_kubernetes_pod_node_name]
        action: replace
        target_label: node
{% endif %}
Relabel configs in plain English: Relabel configs transform labels before scraping. They act as a pipeline: each rule takes source labels, applies a regex, and either keeps/drops the target or replaces a label value. Rules 1 acts as a filter (only scrape annotated pods). Rules 2-3 configure how Prometheus reaches the pod. Rules 4-7 add metadata labels to every scraped metric.

AI Hardening (Epic 4)

Epic 4 makes the AI RCA (Root Cause Analysis) pipeline more reliable and observable. The triage service calls Ollama (an LLM) to analyze alerts, and LLMs are inherently unpredictable. These three stories add guardrails.

AI-01: Circuit Breaker for Ollama Calls

What is a Circuit Breaker?

A circuit breaker is a software pattern borrowed from electrical engineering. It prevents a failing dependency from being called repeatedly, which would waste resources and cascade failures. It has three states:

StateBehaviorTransitions To
CLOSED (normal)Requests pass through to Ollama normally. Failures are counted.OPEN (when failure count reaches threshold)
OPEN (tripped)All requests are immediately rejected without calling Ollama. Returns a fallback response.HALF_OPEN (after a cooldown period expires)
HALF_OPEN (testing)One test request is allowed through. If it succeeds, the breaker resets. If it fails, it trips again.CLOSED (on success) or OPEN (on failure)

The CircuitBreaker Class Flow

class CircuitBreaker:
    """
    States: CLOSED -> OPEN -> HALF_OPEN -> CLOSED (or back to OPEN)

    Configuration:
      failure_threshold: 3       # Trip after 3 consecutive failures
      recovery_timeout: 60       # Wait 60s before trying HALF_OPEN
      half_open_max_calls: 1     # Allow 1 test call in HALF_OPEN
    """

    def __init__(self, failure_threshold=3, recovery_timeout=60):
        self.state = "CLOSED"
        self.failure_count = 0
        self.last_failure_time = None
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout

    async def call(self, func, *args, **kwargs):
        # --- OPEN state: reject immediately ---
        if self.state == "OPEN":
            if self._cooldown_expired():
                self.state = "HALF_OPEN"      # Try one request
            else:
                raise CircuitBreakerOpenError("Circuit breaker is OPEN")

        try:
            result = await func(*args, **kwargs)

            # Success: reset counters
            if self.state == "HALF_OPEN":
                self.state = "CLOSED"          # Recovery confirmed
            self.failure_count = 0
            return result

        except Exception as e:
            self.failure_count += 1
            self.last_failure_time = time.time()

            if self.failure_count >= self.failure_threshold:
                self.state = "OPEN"            # Trip the breaker
                self._emit_metric("circuit_breaker_tripped_total")

            raise

    def _cooldown_expired(self):
        return (time.time() - self.last_failure_time) > self.recovery_timeout

When NEEDS_HUMAN_REVIEW Fires

When the circuit breaker is OPEN and a new alert arrives, the triage service cannot call Ollama for AI analysis. Instead:

  1. The circuit breaker raises CircuitBreakerOpenError
  2. The triage service catches this and sets the verdict to NEEDS_HUMAN_REVIEW
  3. The email sent to the on-call engineer includes: the raw alert data, the MCP context (metrics/logs/traces), and a note that AI analysis was unavailable
  4. A Prometheus metric triage_needs_human_review_total is incremented
  5. When the circuit breaker transitions back to CLOSED, AI analysis resumes automatically

Retry with Exponential Backoff

Before the circuit breaker trips, each individual Ollama call uses retry with exponential backoff:

async def call_ollama_with_retry(prompt, max_retries=3):
    """
    Retry pattern: 1s, 2s, 4s (exponential backoff)
    Each failure counts toward the circuit breaker threshold.
    """
    for attempt in range(max_retries):
        try:
            return await circuit_breaker.call(ollama_client.generate, prompt)
        except CircuitBreakerOpenError:
            raise  # Don't retry if circuit is open
        except Exception as e:
            if attempt == max_retries - 1:
                raise  # Final attempt failed
            delay = 2 ** attempt  # 1s, 2s, 4s
            logger.warning(f"Ollama call failed (attempt {attempt+1}), retrying in {delay}s: {e}")
            await asyncio.sleep(delay)
Interaction between retry and circuit breaker: Each failed attempt increments the circuit breaker's failure counter. With 3 retries per call and a failure threshold of 3, a single completely-failed call is enough to trip the breaker. This is intentional — if Ollama is down, we want to fail fast on subsequent calls.

AI-02: JSON Schema Enforcement for LLM Output

Why format:json Matters

LLMs produce free-form text by default. The triage service needs structured output: a severity, a root cause, and recommended actions. Without enforcement, the LLM might return prose, markdown, or malformed JSON, causing parse failures.

Ollama supports format: "json" in the API request, which constrains the model's output to valid JSON. But valid JSON is not enough — we also need the right schema.

The Pydantic Schema

from pydantic import BaseModel, Field
from typing import List, Optional
from enum import Enum

class Severity(str, Enum):
    CRITICAL = "critical"
    WARNING = "warning"
    INFO = "info"

class Verdict(str, Enum):
    ROOT_CAUSE_FOUND = "root_cause_found"
    INCONCLUSIVE = "inconclusive"
    FALSE_POSITIVE = "false_positive"

class TriageResult(BaseModel):
    """Schema for LLM triage output. Validated with Pydantic after every Ollama call."""
    verdict: Verdict = Field(description="The triage verdict")
    severity: Severity = Field(description="Assessed severity level")
    root_cause: str = Field(description="Identified root cause or best hypothesis")
    evidence: List[str] = Field(description="Supporting evidence from metrics/logs/traces")
    recommended_actions: List[str] = Field(description="Concrete steps to remediate")
    confidence: float = Field(ge=0.0, le=1.0, description="Confidence score 0-1")
    summary: str = Field(description="One-sentence summary for email subject")

Validation Flow

async def analyze_alert(alert_data, mcp_context):
    prompt = build_prompt(alert_data, mcp_context)

    # Call Ollama with JSON format enforcement
    response = await call_ollama_with_retry(
        prompt=prompt,
        format="json"                    # Ollama constrains output to valid JSON
    )

    # Parse and validate against Pydantic schema
    try:
        result = TriageResult.model_validate_json(response)
    except ValidationError as e:
        logger.warning(f"LLM output failed schema validation: {e}")
        # Retry once with a more explicit prompt
        retry_prompt = f"{prompt}\n\nYou MUST respond with valid JSON matching this schema:\n{TriageResult.model_json_schema()}"
        response = await call_ollama_with_retry(prompt=retry_prompt, format="json")

        try:
            result = TriageResult.model_validate_json(response)
        except ValidationError:
            # Double failure: return INCONCLUSIVE
            return build_inconclusive_result(alert_data, mcp_context)

    return result

INCONCLUSIVE Verdict Handling

When the LLM fails to produce a valid schema-conforming response after two attempts, the system returns an INCONCLUSIVE verdict:

  • The email still goes out with all available MCP context (metrics, logs, traces)
  • The email subject is prefixed with [INCONCLUSIVE]
  • The body includes a note: "AI analysis could not produce a structured result. Manual review required."
  • A metric triage_inconclusive_total is incremented

Double-Failure Fallback

def build_inconclusive_result(alert_data, mcp_context):
    """Fallback when LLM output fails validation twice."""
    return TriageResult(
        verdict=Verdict.INCONCLUSIVE,
        severity=Severity.WARNING,
        root_cause="AI analysis failed to produce structured output",
        evidence=["LLM response did not conform to expected schema after 2 attempts"],
        recommended_actions=[
            "Review the alert manually using the attached MCP context",
            "Check Ollama health: kubectl logs -n ai deploy/ollama",
            "Check triage service logs for the full LLM response"
        ],
        confidence=0.0,
        summary=f"[INCONCLUSIVE] Alert: {alert_data.get('alertname', 'unknown')}"
    )
