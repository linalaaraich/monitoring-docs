Kubernetes — The Big Picture
Before any details: a one-page mental model of what Kubernetes is, why it exists, and the universe of things it touches. Come back to this tab whenever the details start to feel disconnected from each other.
What it is, in one sentence

Kubernetes (K8s) is a system that takes a description of how you want your applications to run and continuously works to make reality match that description, on top of a cluster of machines you give it.

That's it. Everything else is detail. The "description" is YAML files. The "continuously works" is a swarm of small programs called controllers. The "cluster of machines" is anywhere from one laptop VM to thousands of cloud servers.

💡
The shift in thinking: with Docker/Compose you write commands like "start this container, restart it if it crashes". With Kubernetes you write "three copies of this container should always exist" and the system figures out where, how, and what to do when something breaks. This is called declarative (vs imperative). It's the single most important idea in K8s.
Why it exists — the short history
  1. Bare metal: one app per server. Wasteful, slow to provision, painful failover.
  2. Virtual machines: many isolated environments per server. Better utilization but each VM is heavy (full OS).
  3. Containers (Docker): lightweight, fast to start, share the host kernel. Solved packaging but not operations at scale.
  4. Container orchestration: how do you run thousands of containers across hundreds of machines, restart them when they fail, route traffic to healthy ones, do rolling upgrades, manage storage, secrets, networking, all automatically? Google had been doing this internally for ~15 years with a system called Borg. They donated a public version called Kubernetes to the CNCF in 2014. Everyone agreed it solved the problem and it became the standard.

The 5 ideas you must internalize first

If you walk away with only these five things, you'll understand 80% of K8s discussions.

① Declarative state

You write a YAML file saying what you want. K8s is responsible for making it true. You don't say "create this pod", you say "this pod should exist". If it gets deleted, K8s recreates it. If you change the YAML, K8s reconciles to the new desired state.

② Reconciliation loops

Inside the cluster, dozens of small programs called controllers are running, each watching one type of object. Each controller is just a loop: while true { compare desired vs actual, take a step toward desired }. That's the whole engine.

③ Everything is an object

Pods, services, configmaps, secrets, even nodes and namespaces — they're all rows in a database (etcd) accessed through one API (kube-apiserver). kubectl get X works for any X. Every object has the same metadata shape (name, namespace, labels, annotations).

④ Labels and selectors, not names

Things connect to each other by matching labels, not by hard-coded names. A Service finds its Pods by saying "give me everything labeled app=spring-boot". This is why you can replace pods freely without breaking anything.

⑤ Controllers compose

A Deployment controller creates ReplicaSets. A ReplicaSet controller creates Pods. A Pod is scheduled by the scheduler. A kubelet runs the container. None of them know about each other directly — they all communicate through the API server. Loose coupling, all the way down.

When NOT to use Kubernetes

K8s is powerful but it has real cost. Be honest about whether you need it.

🚫 Don't use K8s when…
  • You have 1–3 services and one VM works fine.
  • Your team is one person and you're shipping a side project.
  • Your workload is a single long-running binary with no scaling needs.
  • You don't have someone willing to learn it (the operational burden is real).
  • You're optimizing for cost — K8s control plane is overhead.
✅ Do use K8s when…
  • You have many services that need to scale independently.
  • You need self-healing, rolling updates, blue/green or canary deploys.
  • Multiple teams ship to shared infrastructure.
  • You want a portable layer between cloud providers.
  • You need a uniform substrate for observability, secrets, networking, deploys.
  • You're learning it deliberately (the case for this project).
🎯
For our project, the reason is the third one: we want a portable, declarative way to express the same observability target on a laptop, on EC2, and (later) in CIRES private cloud. K8s gives us that. We're not chasing autoscaling — we're chasing reproducibility and skill-building.
Declarative vs imperative — side by side

This is the single biggest mindset shift coming from Docker/Compose. Look at the same task done both ways:

⚡ Imperative (Docker)

"Do these steps in order." If a step fails, the system doesn't know how to recover. If you re-run, you might get a different result. State lives in your head.

docker run -d --name web --restart=always \
  -p 80:80 -e ENV=prod nginx:1.27
docker stop web
docker rm web
docker run -d --name web --restart=always \
  -p 80:80 -e ENV=prod nginx:1.28   # upgrade
# if this crashes mid-way, what state are you in?
📜 Declarative (K8s)

"Here's what should exist." The system figures out the steps. Re-running is a no-op if state already matches. State lives in git.

apiVersion: apps/v1
kind: Deployment
metadata: { name: web }
spec:
  replicas: 1
  selector: { matchLabels: { app: web } }
  template:
    metadata: { labels: { app: web } }
    spec:
      containers:
      - name: nginx
        image: nginx:1.28
        ports: [{ containerPort: 80 }]
# kubectl apply -f web.yaml
# Re-run safely. Roll back with one command.
💡
Subtle but important: the YAML doesn't say "create" or "update". It says "this should exist". kubectl apply is the same command whether the object is brand new, already correct, or needs a change. The system figures it out by comparing your YAML to what's in etcd.
The triangle that runs everything

If you remember nothing else about K8s internals, remember this triangle:

┌────────────────────┐ │ kube-apiserver │ ← the only door in └─────────┬──────────┘ │ reads/writes │ watches for changes │ ┌───────────────────┼───────────────────┐ ▼ ▼ ┌───────────────┐ ┌─────────────────────┐ │ etcd │ │ Controllers │ │ (the truth) │ │ (the workers, ~30 │ └───────────────┘ │ reconcile loops) │ └─────────────────────┘

Three pieces, three jobs:

  • etcd stores objects. Just a key-value store. It doesn't know what a Pod is — it just stores the JSON for any key under /registry/pods/....
  • kube-apiserver is a REST/gRPC front door over etcd. It validates schema, runs admission policies, enforces RBAC. Nothing else in the cluster talks to etcd directly — only through the API server.
  • Controllers are independent loops. They watch the API server (long-poll for changes) and react. Each owns one object kind. Together they implement the entire behavior of K8s.

Everything else — kubelet, kube-proxy, the scheduler — is also "just a controller" in disguise. The kubelet watches Pods assigned to its node. The scheduler watches Pods with no node. The Deployment controller watches Deployments. Same pattern, different scope.

Common misconceptions to clear up now

"Kubernetes runs my containers."  No — kubelet tells containerd to run them. K8s is the orchestrator above; containerd is the runtime below. K8s itself doesn't touch a container.

"Kubernetes is Docker for clusters."  No — Docker is a packaging + runtime tool. K8s removed Docker as a runtime in 1.24 (it uses containerd directly). K8s and Docker live at different layers.

"You need YAML to use K8s."  Yes, but you can generate it. Helm templates, Kustomize overlays, kubectl create deployment foo --image=bar --dry-run=client -o yaml, CDK8s — many ways to avoid hand-writing it.

"Pods are like containers."  Almost — a Pod is one or more containers sharing a network namespace. Most pods have one container, but the Pod abstraction lets you add sidecars (log shippers, proxies, init steps) that share the same IP.

"K8s gives you HA automatically."  No — it gives you the tools for HA (replicas, probes, anti-affinity, PDBs), but you still have to configure them. A 1-replica Deployment is not HA.

"More replicas = more performance."  Only if your app is stateless and your bottleneck is CPU/concurrency. If the bottleneck is the database or a shared lock, replicas don't help.

"K8s can run any workload."  It can schedule any workload, but stateful systems (databases, message queues) take significant additional setup (StatefulSets, persistent storage, operators). Stateless web apps are the easy case.

Core Concepts — The Object Model
Everything in Kubernetes is an object: a row in etcd with a kind, a name, labels, and a spec. Here are the kinds you'll meet first, grouped by what they do. Read this once, then come back when a specific one confuses you.
Workload objects — "what should run"

These describe the things that actually execute code.

Podthe atomic unit
A pod is one or more containers that share a network namespace (same IP, can talk on localhost) and optionally storage volumes. You almost never create pods directly — you create a higher-level object (Deployment) that creates pods for you. Why "pod" and not "container"? So you can run helper sidecars (e.g. a log shipper) in the same network namespace as your main container.
ReplicaSetN copies of a pod
Says "make sure exactly N pods matching this template exist". You also rarely create these directly — Deployment creates them for you. Their job is to maintain count.
Deploymentthe workhorse
The thing you actually write for stateless apps. It manages a ReplicaSet, which manages Pods. Adds rolling updates, rollback, pause/resume, history. 95% of "running an app on K8s" is writing a Deployment + a Service.
StatefulSetstable identity per pod
Like a Deployment but pods get stable, ordered names (db-0, db-1, db-2) and stable storage (each gets its own PVC that follows it across restarts). Use for databases, message queues, anything where pod identity matters.
DaemonSetone pod per node
Runs exactly one copy of a pod on every node (or a labeled subset). Used for node-level agents: log collectors, metrics exporters, CNI plugins, OTel Collector. New node joins → DaemonSet automatically schedules a pod on it.
Jobrun to completion
Runs a pod (or several) until they exit successfully. Use for batch tasks, migrations, backups.
CronJobscheduled Job
A Job that runs on a cron schedule. Like cron but with K8s pod isolation, retries, and history.
Networking objects — "how do things talk"
Servicestable virtual IP
Pods come and go, IPs change. A Service is a stable virtual IP + DNS name that load-balances across all pods matching a label selector. Four types:
  • ClusterIP (default) — only reachable inside the cluster.
  • NodePort — exposed on a port on every node.
  • LoadBalancer — provisions an external load balancer (cloud-only, or k3s ServiceLB).
  • ExternalName — DNS alias to an external host (use for "point to RDS").
IngressHTTP routing
Layer-7 (HTTP/HTTPS) routing rules: "/api → spring-service, / → frontend-service". Needs an Ingress Controller (NGINX, Traefik, Kong, etc.) actually running in the cluster to enforce the rules. Ingress is just the spec; the controller is the worker.
NetworkPolicyfirewall rules
Pod-to-pod firewall rules expressed in YAML: "pods labeled app=db only accept traffic from pods labeled tier=backend". Requires a CNI that enforces them (Calico, Cilium do; Flannel doesn't).
Configuration & secrets
ConfigMapnon-secret config
Key-value pairs for configuration. Mounted into pods as environment variables, files, or command-line args. Use for app config, scrape targets, dashboard JSON.
Secretsensitive config
Same shape as ConfigMap but for sensitive values. Stored base64-encoded (not encrypted by default — see EncryptionConfiguration for at-rest encryption). Mounted as env vars or files. Tools like Sealed Secrets or External Secrets Operator exist for git-safe secret management.
Storage
Volumepod-scoped storage
A directory accessible to a pod. Lives as long as the pod does. Comes in many "types": emptyDir (scratch), hostPath (node disk), and the more useful kinds below.
PersistentVolume (PV)cluster-scoped storage
A piece of storage in the cluster — a chunk of an EBS volume, an NFS export, etc. Lives independently of any pod. Created by an admin or auto-provisioned.
PersistentVolumeClaim (PVC)a request for storage
A pod says "I need 10Gi of storage with class fast-ssd". K8s finds or provisions a PV that matches and binds them. The pod mounts the PVC; the PV moves with the pod across reschedules.
StorageClassstorage profile
A "type" of storage (e.g. gp3-ssd, standard-hdd) and the driver that provisions it. Cloud distros come with one default; on-prem you install one (e.g. local-path-provisioner on k3s, Longhorn, Rook/Ceph).
Organization & identity
Namespacevirtual cluster
A way to slice a cluster into independent environments — prod, staging, per-team, per-app. RBAC, quotas, NetworkPolicies are mostly namespace-scoped. Cluster-wide objects (Nodes, PVs, ClusterRoles) are not.
Label / Annotationmetadata
Labels are short key-value pairs used for selection (app=spring-boot, tier=backend). Annotations are for metadata that tools read but K8s ignores (e.g. prometheus.io/scrape: "true"). Labels are queryable; annotations are not.
ServiceAccountidentity for pods
Every pod runs as a ServiceAccount. The ServiceAccount has tokens it uses to talk to the API server (e.g. for an operator that needs to read pods). Distinct from a human user.
Role / RoleBindingRBAC
A Role says "this set of verbs (get, list, create, delete) on these resources (pods, configmaps)". A RoleBinding grants that Role to a ServiceAccount or user. ClusterRole / ClusterRoleBinding are the cluster-wide versions.
Extensibility
CRDcustom object kind
CustomResourceDefinition lets you teach the API server a new object kind. Once registered, kubectl get mything works just like kubectl get pods. The object exists in etcd; it does nothing until a controller acts on it.
OperatorCRD + controller
A pattern: define a CRD (e.g. PostgresCluster) and write a controller that watches for those objects and reconciles them into real workloads (StatefulSets, Services, PVCs, Secrets). It's how you encode operational knowledge into the cluster. Examples: Prometheus Operator, cert-manager, OpenTelemetry Operator.
🧭
Anchor concepts for our project: Deployment + Service + Ingress for the apps; DaemonSet for the in-cluster OTel Collector; ConfigMap for prom scrape config and Grafana dashboards; Secret for RDS credentials; PVC for any stateful storage; CRD/Operator if we adopt the OpenTelemetry Operator. That's basically all you need to build the migration.

A Deployment — what you actually write
apiVersion: apps/v1
kind: Deployment
metadata:
  name: spring-boot
  namespace: app
  labels:
    app: spring-boot
spec:
  replicas: 3
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # one extra pod during rollout
      maxUnavailable: 0    # never go below desired count
  selector:
    matchLabels:
      app: spring-boot
  template:                # this is the Pod template
    metadata:
      labels:
        app: spring-boot   # MUST match selector above
    spec:
      containers:
      - name: app
        image: cires/spring-boot:1.4.2     # ALWAYS pin a version
        ports:
        - containerPort: 8080
        envFrom:
        - configMapRef: { name: spring-config }
        - secretRef: { name: rds-credentials }
        resources:
          requests: { cpu: 200m, memory: 512Mi }
          limits:   { cpu: 1000m, memory: 1Gi }
        livenessProbe:
          httpGet: { path: /actuator/health/liveness, port: 8080 }
          initialDelaySeconds: 30
        readinessProbe:
          httpGet: { path: /actuator/health/readiness, port: 8080 }
          initialDelaySeconds: 10

Notice: the selector.matchLabels and template.metadata.labels must match. This is how the Deployment controller knows which Pods belong to it.

A Service that fronts the Deployment
apiVersion: v1
kind: Service
metadata:
  name: spring-boot
  namespace: app
spec:
  type: ClusterIP                    # internal only (default)
  selector:
    app: spring-boot                 # finds pods by label
  ports:
  - name: http
    port: 80                         # the Service port
    targetPort: 8080                 # the container port
    protocol: TCP

Now any pod in the cluster can reach this with http://spring-boot.app.svc.cluster.local (or just http://spring-boot from inside the same namespace). K8s gives it a virtual IP and load-balances across all healthy pods matching app=spring-boot.

A Secret (almost identical to ConfigMap)
apiVersion: v1
kind: Secret
metadata:
  name: rds-credentials
  namespace: app
type: Opaque
stringData:                          # plain text (k8s base64-encodes for you)
  DB_USERNAME: app_user
  DB_PASSWORD: "super-secret-pw"
# Use as envFrom: [{ secretRef: { name: rds-credentials } }]
⚠️
Secrets are NOT encrypted by default — they're base64-encoded. Anyone with get secrets RBAC can read them in plain text. For real protection: enable EncryptionConfiguration on the API server, or use Sealed Secrets / External Secrets Operator / SOPS.
A StatefulSet (for databases & queues)
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
  namespace: db
spec:
  serviceName: postgres              # required: a headless Service
  replicas: 3
  selector:
    matchLabels: { app: postgres }
  template:
    metadata:
      labels: { app: postgres }
    spec:
      containers:
      - name: postgres
        image: postgres:16.2
        ports: [{ containerPort: 5432 }]
        volumeMounts:
        - name: data
          mountPath: /var/lib/postgresql/data
  volumeClaimTemplates:              # each pod gets its own PVC
  - metadata: { name: data }
    spec:
      accessModes: [ "ReadWriteOnce" ]
      storageClassName: fast-ssd
      resources:
        requests: { storage: 20Gi }

Pods are named postgres-0, postgres-1, postgres-2 (stable!). Each gets its own PVC: data-postgres-0, data-postgres-1, etc. If postgres-1 dies, it comes back as postgres-1 with the same disk.

Labels & selectors — the secret sauce

Labels are how everything in K8s connects. They're short key=value pairs on every object. Selectors are queries against them. You will use them constantly, so internalize the patterns:

# Common label conventions (the "recommended labels")
metadata:
  labels:
    app.kubernetes.io/name: spring-boot       # the app name
    app.kubernetes.io/instance: prod-1         # this specific instance
    app.kubernetes.io/version: "1.4.2"
    app.kubernetes.io/component: backend
    app.kubernetes.io/part-of: cires-platform
    app.kubernetes.io/managed-by: helm

# Selectors used in Service / NetworkPolicy / etc:
selector:
  matchLabels:
    app: spring-boot

# More expressive form:
selector:
  matchExpressions:
  - { key: tier, operator: In, values: [backend, middleware] }
  - { key: env, operator: NotIn, values: [dev] }

# Selecting from kubectl:
kubectl get pods -l app=spring-boot,tier=backend
kubectl get pods -l 'env in (prod, staging)'
kubectl delete pods -l app=spring-boot       # all spring-boot pods
💡
Why labels and not names: Pod names change (they include random hashes). Labels don't. A Service's selector remains valid even as Pods are created and destroyed. This decoupling is what makes self-healing possible — you can replace pods freely and references keep working.
Architecture — What's Inside a Cluster
A Kubernetes cluster has two kinds of machines: the control plane (the brain) and worker nodes (where your apps actually run). On a small cluster they can be the same machine. Here's everything that runs and what it does.
🧠 Control Plane (the brain)
kube-apiserver etcd kube-scheduler kube-controller-manager cloud-controller-manager
💪 Worker Node (one per machine)
kubelet kube-proxy container runtime (containerd) CNI plugin
🧩 Standard Add-ons (deployed as pods)
CoreDNS metrics-server Ingress Controller Storage provisioner Dashboard (optional)
Control plane components
CORE kube-apiserver

The only door into the cluster. Every kubectl command, every controller, every kubelet, talks to it via REST. It validates requests, enforces RBAC, runs admission webhooks, and reads/writes etcd. If the API server is down, nothing in the cluster can be changed (existing pods keep running).

Stateless — you can run multiple replicas behind a load balancer for HA. State lives in etcd.

CORE etcd

The cluster's database. A distributed key-value store (Raft consensus). Every K8s object you've ever created is a key in etcd under /registry/.... Lose etcd, lose the cluster's memory.

Production clusters run 3 or 5 etcd members (odd numbers for quorum). For HA, etcd is the hard part — backup is non-negotiable. k3s replaces etcd with SQLite by default for single-node, or embedded etcd / external SQL for HA.

CORE kube-scheduler

Decides where pods run. Watches for pods with no node assignment, scores every node against constraints (resources, labels, affinity, taints, topology), picks the best one, writes the choice back to the API server. Doesn't actually run the pod — that's the kubelet's job.

Stateless. The decision logic is pluggable; you can write a custom scheduler if you need to.

CORE kube-controller-manager

The reconciliation engine. A single binary running ~30 controllers as goroutines: Deployment controller, ReplicaSet controller, Node controller, Service controller, Endpoint controller, Job controller, ServiceAccount controller, etc. Each watches one type of object and drives actual state toward desired state.

This is where the "K8s magic" actually happens. When you create a Deployment, the Deployment controller notices and creates a ReplicaSet. The ReplicaSet controller notices and creates Pods. The scheduler notices unscheduled Pods and assigns them to nodes. The kubelet notices a pod assigned to its node and starts the container. Nobody told anybody to do anything — they all just react to the API server.

OPTIONAL cloud-controller-manager

The bridge between K8s and a cloud provider's APIs. Translates "Service type LoadBalancer" into "create an AWS ELB", "Node deleted" into "release this EC2 instance", etc. Only present on cloud-managed clusters or self-managed with cloud integration. Not relevant for k3s on a single VM.

Worker node components
CORE kubelet

The agent on every node. Watches the API server for pods assigned to its node, then talks to the container runtime to actually start them. Reports node health and pod status back. Runs liveness/readiness probes. Mounts volumes. If kubelet dies on a node, that node goes "NotReady" and the scheduler stops sending pods there.

Configured via /var/lib/kubelet/config.yaml and a kubeconfig that points to the API server.

CORE kube-proxy

Implements Services on each node. Watches Services and their Endpoints, then writes iptables (or IPVS, or eBPF) rules so traffic to a Service IP gets load-balanced to actual pod IPs. Runs as a DaemonSet on every node. Some modern CNIs (Cilium) replace kube-proxy entirely with eBPF.

You'll see kube-proxy logs blame "the network is broken" when really an iptables rule is missing — knowing it exists matters.

CORE Container runtime

The thing that actually starts containers. kubelet talks to it via the CRI (Container Runtime Interface). Today this is almost always containerd (lightweight, used by k3s, EKS, etc.). Older clusters used Docker via dockershim — that was removed in K8s 1.24.

You can shell into a node and use crictl ps to see what's running, just like docker ps but for containerd.

CORE CNI plugin

Pod networking. The Container Network Interface is a spec; the plugin (Flannel, Calico, Cilium, Weave, etc.) is the implementation. It gives every pod its own IP, sets up routing between nodes, and (depending on plugin) enforces NetworkPolicies.

The K8s networking model has three rules: every pod gets a unique IP, pods can talk to each other without NAT, and nodes can talk to pods without NAT. The CNI's job is to make those rules true on whatever underlying network you have.

Standard add-ons (deployed as regular workloads)
ADD-ON CoreDNS

In-cluster DNS. Resolves service names: my-service.my-namespace.svc.cluster.local → ClusterIP. Without it, pods can't find each other by name. Runs as a Deployment with 2 replicas by default. Bundled in essentially every distro.

ADD-ON metrics-server

Lightweight CPU/memory aggregator. Powers kubectl top pods and HorizontalPodAutoscaler. Not Prometheus — much simpler, in-memory only. You install it separately on most distros (k3s bundles it).

ADD-ON Ingress Controller

An actual program (NGINX, Traefik, Kong, HAProxy, etc.) that reads Ingress objects and translates them into reverse-proxy config. Needed if you want HTTP routing into the cluster. Each distro picks a default; you can replace it.

ADD-ON Storage provisioner

Watches for PVCs and creates PVs. The cloud distros use cloud-CSI drivers (EBS, GCE PD, Azure Disk). On bare metal, common picks are local-path-provisioner (k3s default), Longhorn, OpenEBS, Rook/Ceph.

How it all fits together

Imagine you run kubectl apply -f deployment.yaml. Here's what happens, end to end:

1

kubectl → API server

Your kubectl client authenticates (cert in ~/.kube/config) and POSTs the Deployment YAML to the API server.

2

API server: auth → admission → validation → persist

RBAC check, admission webhooks run (e.g. inject sidecars), the object is validated against its schema, then written to etcd. The API server now has a Deployment object.

3

Deployment controller wakes up

It was watching for Deployments. It sees the new one, creates a ReplicaSet, POSTs it to the API server.

4

ReplicaSet controller wakes up

It sees a new ReplicaSet wanting N pods. It creates N Pod objects via the API server. Each pod is created with no node assigned.

5

Scheduler wakes up

It sees pods with empty spec.nodeName. It scores nodes, picks one per pod, and writes the assignment back to the API server.

6

kubelet on the chosen node wakes up

It was watching for pods assigned to its node. It pulls the container image (via containerd), sets up volumes and networking (via the CNI), and starts the container.

7

Status flows back

kubelet reports the pod is Running. The API server updates the Pod object's status. kubectl get pods reflects it.

Notice: nothing was given a direct command. Every step was a controller noticing a state change in the API server and reacting. This is the reconciliation loop in action.

Leader election & HA

In a multi-server (HA) setup, there are multiple instances of kube-controller-manager and kube-scheduler. Only one is "leader" at a time. They use a Lease object in the kube-system namespace to coordinate — the current leader renews the lease every few seconds; if it dies, another instance grabs it. The API server itself is stateless and runs active-active behind a load balancer. etcd uses Raft consensus for HA (3, 5, or 7 members; odd numbers for quorum).

How K8s Actually Works
The previous tab listed the components. This one shows the patterns those components use to do their jobs: reconciliation, networking, storage, security, health checks. These are the patterns that repeat everywhere.
The reconciliation loop pattern

Every controller in the cluster follows the same shape:

while true {
  desired = read_from_api_server()       // what should exist
  actual  = observe_real_world()         // what does exist
  diff    = compute_difference(desired, actual)
  if diff != empty {
    take_one_step_toward(desired)        // create, update, delete
  }
  sleep_or_wait_for_event()
}

Three properties fall out of this:

  • Idempotent. Running the loop ten times has the same effect as running it once. You can never "double-create" something.
  • Self-healing. If you delete a Pod managed by a Deployment, the ReplicaSet controller sees actual < desired and creates another. You can't keep things broken without changing the desired state.
  • Eventually consistent. Changes propagate through the system asynchronously. kubectl apply returns instantly; the actual change can take seconds to manifest.
The networking model

K8s has three rules that any conformant network must satisfy:

  1. Every pod gets its own IP.
  2. Pods can talk to each other directly without NAT.
  3. Nodes can talk to pods directly without NAT.

The CNI plugin makes these true. Different CNIs do it differently:

  • Flannel (k3s default): VXLAN overlay. Each node gets a pod-IP subnet, packets between nodes are wrapped in UDP. Simple, fast, no NetworkPolicy.
  • Calico: BGP-based, no overlay if your network supports it. Enforces NetworkPolicies. Common production choice.
  • Cilium: eBPF-based. Replaces kube-proxy. Very fast, very featureful, slightly more complex to operate.
How a Service actually works

You create a Service of type ClusterIP. The API server allocates an unused IP from the service CIDR. The Endpoints controller watches the Service's selector and writes the matching pod IPs into an Endpoints object. kube-proxy on every node watches Endpoints and writes iptables rules:

# Roughly, this is what kube-proxy installs:
-A KUBE-SERVICES -d 10.96.0.42/32 -p tcp --dport 80 -j KUBE-SVC-XYZ
-A KUBE-SVC-XYZ -m statistic --mode random --probability 0.333 -j KUBE-SEP-A   # pod 1
-A KUBE-SVC-XYZ -m statistic --mode random --probability 0.500 -j KUBE-SEP-B   # pod 2
-A KUBE-SVC-XYZ -j KUBE-SEP-C                                                  # pod 3
-A KUBE-SEP-A   -p tcp -j DNAT --to-destination 10.244.1.5:80                  # actual pod IP
-A KUBE-SEP-B   -p tcp -j DNAT --to-destination 10.244.2.7:80
-A KUBE-SEP-C   -p tcp -j DNAT --to-destination 10.244.3.9:80

That's the entire "magic" of a Service: random load balancing via iptables NAT. No proxy process in the data path. When pods change, kube-proxy rewrites the rules.

DNS

CoreDNS gives every Service a DNS name: <service-name>.<namespace>.svc.cluster.local. Pods are configured (via /etc/resolv.conf injected by kubelet) to use CoreDNS. So inside a pod, curl http://prometheus.monitoring resolves the Prometheus Service in the monitoring namespace and gets load-balanced to a real pod.

The storage flow

A pod that needs persistent storage:

1

Pod references a PVC

In its spec: volumes: [{ name: data, persistentVolumeClaim: { claimName: my-pvc }}].

2

PVC requests storage from a StorageClass

The PVC says "I need 10Gi from class fast-ssd".

3

The provisioner creates a PV

The CSI driver for that StorageClass talks to the underlying storage system (AWS EBS API, NFS server, etc.) and creates a real volume. Wraps it in a PV object.

4

K8s binds PVC to PV

Now they're paired one-to-one.

5

kubelet attaches and mounts

On the node where the pod is scheduled, kubelet attaches the volume and mounts it into the container's filesystem at the path the pod specified.

If the pod is rescheduled to another node, the volume detaches from the old node and re-attaches to the new one. The pod keeps its data.

The security model
Authentication (who are you)

Three modes:

  • Client certificates — what kubectl uses by default. Your ~/.kube/config has a cert signed by the cluster CA.
  • Bearer tokens — what pods use. Each ServiceAccount has a token mounted at /var/run/secrets/kubernetes.io/serviceaccount/token.
  • OIDC — for human users via SSO providers (Google, Okta, Keycloak).
Authorization (what can you do)

RBAC. A Role lists allowed verbs on resources. A RoleBinding grants the Role to a user or ServiceAccount. Example:

Admission control

After auth/RBAC, the request passes through admission webhooks. These can mutate the object (e.g. inject a sidecar) or reject it (e.g. block privileged containers). This is where most security policy lives in modern clusters.

Health probes

kubelet runs three kinds of probes against every pod:

  • Liveness probe — "is this container still healthy?". Failure → restart the container.
  • Readiness probe — "should this container receive traffic right now?". Failure → remove from Service endpoints (no restart).
  • Startup probe — "is the container still starting up?". While failing, the other probes are paused. Use for slow-starting apps.

Probe types: HTTP GET, TCP socket, exec a command in the container. Misconfigured probes are the #1 source of "why is my pod CrashLoopBackOff" pain. Always set them, always test them.

Resource requests and limits

Every container can declare:

  • Requests — what the pod needs to be scheduled. The scheduler reserves this much CPU/memory on the node.
  • Limits — the cap. Exceed CPU limit → throttled. Exceed memory limit → OOMKilled.

If you don't set requests, the scheduler can't make smart decisions. If you don't set limits, a runaway container can starve the node. Set both. Always.

QoS classes

Based on how you set requests and limits, K8s assigns each pod a Quality of Service class. This decides who gets killed first when the node runs out of memory:

ClassHow to get itEviction priority
GuaranteedEvery container has requests == limits for both CPU and memoryKilled last
BurstableAt least one container has requests < limits, or requests set without limitsKilled in the middle
BestEffortNo requests or limits set anywhereKilled first
The scheduler — how it picks a node

The scheduler runs two phases for every unscheduled pod:

  1. Filter (predicates) — eliminate nodes that can't run the pod. Reasons: not enough resources, doesn't tolerate the node's taints, fails nodeSelector / affinity, missing required volume.
  2. Score (priorities) — rank the remaining nodes. Factors: least requested resources, balanced CPU/memory, image already cached locally, topology spread, inter-pod affinity.

Highest score wins. Tie → random. The decision is then written back to the pod's spec.nodeName via the API server.

Influencing the scheduler

  # 1. Hard requirement — won't schedule unless matched
  nodeSelector:
    disktype: ssd

  # 2. More expressive
  affinity:
    nodeAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        nodeSelectorTerms:
        - matchExpressions:
          - key: kubernetes.io/arch
            operator: In
            values: [amd64, arm64]

    # Spread my pods across zones for HA
    podAntiAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
      - weight: 100
        podAffinityTerm:

The pod-to-pod networking walkthrough

Two pods on different nodes want to talk. Here's what actually happens with a typical CNI (Flannel VXLAN):

  1. Pod A on Node 1 (IP 10.42.1.5) sends a packet to Pod B on Node 2 (IP 10.42.2.7).
  2. The packet leaves Pod A's network namespace via a veth pair into Node 1's root network namespace.
  3. Node 1's routing table has a route for 10.42.2.0/24 via the Flannel device (flannel.1).
  4. Flannel wraps the packet in a VXLAN UDP datagram destined for Node 2's IP (a real, routable IP).
  5. The wrapped packet travels through whatever network connects the nodes (a real switch, a cloud VPC, etc.).
  6. Node 2 receives it, unwraps the VXLAN, sees a packet for 10.42.2.7.
  7. Node 2's routing table sends it via veth pair into Pod B's network namespace.
  8. Pod B sees a packet from 10.42.1.5 — no NAT, no rewrite. From its point of view, it just got a packet from another pod.

Calico does the same thing without VXLAN encapsulation if your underlying network can route pod CIDRs directly (BGP). Cilium does it with eBPF programs in the kernel, no iptables. All three satisfy the same K8s networking model.

DNS lookup walkthrough

Inside a pod, you run curl http://prometheus.monitoring. What happens:

  1. libc reads /etc/resolv.conf in the pod (kubelet wrote it). Contains: nameserver 10.96.0.10 (the CoreDNS Service ClusterIP) and search default.svc.cluster.local svc.cluster.local cluster.local.
  2. libc tries to resolve prometheus.monitoring. It appends each search domain in turn: prometheus.monitoring.default.svc.cluster.local (NXDOMAIN), then prometheus.monitoring.svc.cluster.local (hit!).
  3. The DNS query goes to 10.96.0.10:53. iptables rules from kube-proxy redirect this to one of the actual CoreDNS pod IPs.
  4. CoreDNS looks up the Service, returns its ClusterIP, e.g., 10.96.42.7.
  5. libc returns 10.96.42.7 to curl. curl opens a TCP connection.
  6. The packet goes out. iptables (kube-proxy again) rewrites the destination to a random healthy pod IP behind the prometheus Service.
  7. The connection lands on a real prometheus pod.

Two layers of iptables magic (DNS and Service), all transparent to the application. This is why "trace where the traffic actually goes" is a useful K8s debugging skill.

Init containers and sidecars

A Pod can have multiple containers. They share the network namespace (same IP, can talk on localhost) and optionally volumes.

Init containers

Run before the main containers, in order, until each succeeds. Used for: waiting for a dependency, running migrations, copying assets to a shared volume.

spec:
  initContainers:
  - name: wait-for-db
    image: busybox:1.36
    command: ['sh', '-c', 'until nc -z postgres 5432; do sleep 1; done']
  - name: run-migrations
    image: cires/spring-boot:1.4.2
    command: ['java', '-jar', 'app.jar', '--migrate-only']
  containers:
  - name: app
    image: cires/spring-boot:1.4.2
    # only starts after both init containers succeed
Sidecars

Containers that run alongside the main one for the lifetime of the pod. Use cases: log shipper reading the main container's stdout, an OTel agent receiving spans on localhost, a service mesh proxy (Envoy/Linkerd). In K8s 1.29+, sidecars have a special "restartable init container" form so they start before the main and stop after it.

PV lifecycle & access modes

A PV moves through phases: Available → Bound → Released → Deleted/Retained.

  • Available — created but not yet claimed.
  • Bound — matched to a PVC. One-to-one.
  • Released — the PVC was deleted but the PV still exists (waiting for a reclaim decision).
  • Deleted or Retained — depending on the StorageClass's reclaim policy: Delete destroys the underlying storage, Retain keeps it for manual recovery.

Access modes (a PVC requests one):

  • ReadWriteOnce (RWO) — one node can mount it read-write. Most cloud block storage (EBS, etc.).
  • ReadOnlyMany (ROX) — many nodes can mount it read-only.
  • ReadWriteMany (RWX) — many nodes can mount it read-write. Needs a network filesystem (NFS, CephFS, EFS).
  • ReadWriteOncePod (RWOP, K8s 1.27+) — exactly one Pod can mount it. Stricter than RWO.

Debugging recipes

Things will break. Here's where to look first.

"My pod is in Pending forever"
k describe pod <name> | tail -20
# Look at Events. Common causes:
# - "Insufficient cpu/memory" → bump node size or reduce requests
# - "0/3 nodes are available: 3 node(s) had untolerated taint" → add tolerations
# - "no PV available" → no StorageClass or quota issue
# - "FailedScheduling" with no detail → look at scheduler logs
"My pod is CrashLoopBackOff"
k logs <pod>                        # current container
k logs <pod> --previous             # the one that just died
k describe pod <name> | grep -A5 "Last State"
# Look for: Exit Code, Reason: OOMKilled / Error / Completed
# OOMKilled → bump memory limit
# Exit Code 1 → app error, check the previous logs
# Exit Code 137 → SIGKILL, often OOM
# Exit Code 139 → segfault
# Probe failures → check liveness/readiness URLs and timing
"I can't reach my Service"
# 1. Does the Service exist and have endpoints?
k get svc <name>
k get endpoints <name>              # if EMPTY, no pods match the selector
k get pods -l <selector-from-svc>   # do those pods exist and run?

# 2. Are pods Ready (passing readinessProbe)?
k get pods -l app=<name>            # READY column is X/Y

# 3. From inside the cluster, does DNS resolve?
k run -it --rm dnstest --image=busybox --restart=Never -- nslookup <svc-name>

# 4. Can you reach the pod IP directly (skip the Service)?
k run -it --rm test --image=busybox --restart=Never -- wget -qO- <pod-ip>:<port>
"ImagePullBackOff"
k describe pod <name> | grep -A3 "Failed"
# Common causes:
# - Wrong image tag (typo, deleted upstream)
# - Private registry without imagePullSecret
# - Rate-limited (Docker Hub anonymous pulls)
# Fix: check image name, add imagePullSecrets, or pre-load with `crictl pull`
"My ConfigMap/Secret change isn't taking effect"

Pods don't auto-reload mounted ConfigMaps/Secrets. Restart the deployment:

k rollout restart deploy/<name>

For environment variables specifically, you must restart — they're injected at pod start. Mounted file ConfigMaps update in-place but the app has to re-read the file (most don't).

Reproducibility — What It Means & When You Want It
"Reproducible" means: starting from a clean machine and the same inputs, you can reach the same end state. Every time. By anyone. This is harder than it sounds, and K8s gives you both the tools to achieve it and plenty of ways to lose it.
What "reproducible" actually means

The strict definition: given the same source code at git commit X, an empty cloud account, and the same Terraform/Ansible/Helm files, anyone running the same commands gets the same cluster with the same workloads behaving identically.

The loose definition (most teams settle here): the cluster is reproducible enough that we can recover from a disaster, redeploy from scratch, and run the same code in dev and prod without surprises.

Why you want it — and when you don't
✅ You want reproducibility when…
  • The system runs in production and downtime is costly.
  • Multiple people need to deploy and the result must match.
  • You need to recover from a region outage or accidental deletion.
  • You need to audit "what was running on day X".
  • You're moving environments (dev → staging → prod, on-prem → cloud).
  • Compliance requires it (regulated industries).
⚠️ Reproducibility costs you when…
  • You're exploring — pinning everything slows iteration.
  • You're a one-person side project with no recovery requirement.
  • Security patches need to flow in fast — pinning blocks that unless you also automate updates.
  • You haven't decided what the system is yet — locking it down prematurely is wasted effort.

For our project the answer is yes, fully reproducible, because the whole point is to demonstrate "we can deploy this exact stack anywhere on demand". The whole IaC story falls apart if a fresh run produces a different cluster.

The reproducibility ladder

Reproducibility isn't one switch — it's a layered set of decisions. Each layer pinned eliminates a class of drift. From bottom (machine) to top (apps):

1
Infrastructure
Terraform with pinned required_providers versions. Pin AMI IDs (not most_recent = true). Pin instance types, region, AZ. Commit .terraform.lock.hcl.
2
OS & node bootstrap
Ansible with pinned role versions in requirements.yml (use git refs / version tags, never master). Pin package versions where possible.
3
Kubernetes version
Install k3s with INSTALL_K3S_VERSION=v1.30.2+k3s1 (or kubeadm's kubelet=1.30.2-00 kubeadm=1.30.2-00). Never the unpinned curl pipe.
4
CNI / Ingress / Storage
If you install your own CNI or storage class, pin their versions. If you use a distro's bundled defaults, the K8s version pin (rung 3) covers them.
5
Helm charts
Always pass --version 1.2.3 or set chart_version: "1.2.3" in your tooling. Commit your values.yaml to git. Lock with helm dependency update producing a Chart.lock.
6
Container images
Pin by digest: image: nginx@sha256:abc123…. At minimum by tag: nginx:1.27.0. Never :latest. For your own images, build with a versioned tag in CI.
7
Manifests & config
All YAML committed to git. No "I tweaked it in the cluster" — that's drift. Use kubectl diff to detect drift before applying.
8
Secrets
Sealed Secrets, External Secrets Operator, or SOPS-encrypted YAML. Encrypted form goes in git, plaintext never does. The decryption key is the only out-of-band secret.
⚠️
The single biggest reproducibility killer: image: myapp:latest. Every deploy might pull a different version of the image. The pod template hash doesn't change, so K8s thinks nothing changed and won't even roll. Always tag immutably.
Imperative vs declarative — and why GitOps helps

You can manage K8s in two styles:

  • Imperative: kubectl create deployment, kubectl edit, helm upgrade from a developer laptop. Fast for exploring, hostile to reproducibility.
  • Declarative: write YAML files, commit them, apply them. Reproducible and reviewable.

GitOps takes declarative one step further: instead of humans running kubectl apply, a controller in the cluster (ArgoCD or Flux) watches a git repo and continuously syncs. The cluster is what's in git, by definition. Drift is a violation that gets corrected automatically. This is the gold standard for production reproducibility.

🎯
Where we are vs where we'd go: for sprint 2, manual helm install + Ansible is fine. For sprint 3 or beyond, adopting ArgoCD would let us push the declarative discipline all the way and remove "did you remember to redeploy?" as a question.

Concrete examples — each ladder rung in action
Rung 1: Terraform with pinned providers
# providers.tf
terraform {
  required_version = "~> 1.7.0"        # pin Terraform itself
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "= 5.43.0"              # exact version
    }
  }
}

# ec2.tf — pin the AMI explicitly
resource "aws_instance" "k3s" {
  ami           = "ami-0a3c3a20c09d6f377"   # specific AMI ID, not "most recent"
  instance_type = "t3.large"
  # ...
}

# Commit .terraform.lock.hcl to git — it pins provider plugin checksums
⚠️
Anti-pattern: data "aws_ami" "ubuntu" { most_recent = true } — picks whatever AMI is newest at apply time. Two runs a week apart can produce different VMs. Pin the AMI ID once, update it deliberately.

# Helmfile / Ansible / shell — always pass --version

  --version 58.2.1 \                    # pin chart version

Rung 6: Image digests (the strongest pin)
# Tag (good)
image: prom/prometheus:v2.51.0

# Digest (best — guaranteed identical bytes)
image: prom/prometheus@sha256:5c5d5e9c9c6c5a1f8c7e8f1234567890abcdef...

# Get the digest of a tag:
crane digest prom/prometheus:v2.51.0
# or
docker buildx imagetools inspect prom/prometheus:v2.51.0
🚫
Never write image: prom/prometheus (no tag — defaults to :latest) or image: prom/prometheus:latest. The tag :latest is mutable upstream — you'll get whatever the maintainer pushed last, possibly different on every node, possibly different on every restart.
Rung 8: Secrets that live safely in git

Sealed Secrets — you encrypt with a public key, the controller in the cluster decrypts with a private key. The encrypted form is safe to commit:

# Plain Secret (don't commit this)
echo -n 'super-secret-pw' | kubectl create secret generic db \
  --dry-run=client --from-file=password=/dev/stdin -o yaml > secret.yaml

# Seal it
kubeseal --controller-name sealed-secrets-controller \
  -o yaml < secret.yaml > sealed-secret.yaml

# sealed-secret.yaml is now safe to commit. Apply with kubectl apply -f.
# Inside the cluster, sealed-secrets-controller decrypts and creates the real Secret.
Anti-pattern gallery

🚫 kubectl edit deploy/foo in production. Changes the cluster but not git. Now git no longer reflects reality. Drift. Don't.

🚫 helm install without --version. Pulls whatever is "stable" at apply time. Could be a new major version with breaking changes.

🚫 image: myapp:latest. Already covered. Worth saying twice.

🚫 Secrets in plaintext YAML committed to git. Even if your repo is private, anyone who clones it has them forever. Use sealed-secrets, SOPS, or external-secrets.

🚫 "It's only dev, version pins don't matter". Dev that doesn't match prod is the source of "works on my machine" bugs. The cost of pinning is near zero — pay it.

🚫 Hand-edited kubectl apply from a developer laptop with no git trail. That change exists nowhere except in someone's shell history. Will be lost on the next deploy.

Detecting drift

Even with discipline, drift creeps in. Tools to find it:

  • kubectl diff -f manifest.yaml — what would change if I applied this now?
  • ArgoCD — its UI shows "OutOfSync" for any object that doesn't match git, and offers a one-click sync.
  • Flux — same idea, different UX.
  • kube-no-trouble (kubent) — finds deprecated APIs that will break on upgrade.

Use it when: you're shipping production workloads in a single cloud and don't want to operate the control plane. Skip for learning or for "I want to understand what the cloud is hiding."

Use it when: a large org has standardized on it. Not a learning target; you'll meet it if you join a company that uses it.

k3s — Deep Dive
Since k3s is what we'll actually use, here's everything you need to know to operate it confidently: what's inside, how to install it, how it differs from vanilla K8s, how to make it production-grade, and where it fits with our Ansible/Terraform workflow.
What k3s actually is

k3s is a Kubernetes distribution packaged as a single ~70MB Go binary. When you run k3s server, that binary starts an entire control plane (API server, scheduler, controller manager, etcd-or-SQLite) and a kubelet, all as goroutines in one process. When you run k3s agent on another machine, it joins as a worker node.

It's built by Rancher (now SUSE) and is CNCF-certified. The "k3" in the name is a play on "k8s minus 5" — the original goal was to remove enough optional features to fit in 512 MB of RAM. The result is real upstream K8s with sane defaults baked in.

What's bundled (and what to disable)

Out of the box, k3s includes everything you'd otherwise install yourself:

ComponentWhat it doesDisable with
containerdContainer runtime (replaces Docker)Use --container-runtime-endpoint to point at another runtime
FlannelCNI plugin (pod networking)--flannel-backend=none + install your own CNI (Calico, Cilium)
CoreDNSIn-cluster DNS--disable=coredns (rarely needed)
TraefikIngress controller--disable=traefik ← we'll do this to keep Kong
ServiceLB (klipper-lb)LoadBalancer Service implementation (no cloud needed)--disable=servicelb
local-path-provisionerDefault StorageClass for PVCs--disable=local-storage
metrics-serverPowers kubectl top and HPA--disable=metrics-server

Each disabled component is one thing you have to provide yourself. For our project we'll disable Traefik (Kong replaces it) and keep everything else.

How k3s differs from vanilla K8s
  • Single binary, single process — control plane components run as goroutines, not separate processes. journalctl -u k3s shows logs for everything.
  • Default datastore is SQLite for single-node. Vanilla uses etcd always. k3s can also use embedded etcd (HA mode) or external MySQL/Postgres.
  • kine — k3s ships with a shim called kine that exposes any database (SQLite, MySQL, Postgres) as if it were etcd. This is how it can use SQLite without breaking compatibility.
  • No separate kube-proxy DaemonSet — kube-proxy runs inside the k3s process.
  • Storage and ingress bundled by default — vanilla K8s ships with neither.
  • kubeconfig in /etc/rancher/k3s/k3s.yaml, not ~/.kube/config. Symlink or copy it.
  • Static pod manifests at /var/lib/rancher/k3s/server/manifests/ — drop YAML files here and k3s applies them automatically (great for bootstrap).
Installing k3s
Quick install (don't do this in IaC)
curl -sfL https://get.k3s.io | sh -

This pulls the latest, installs the systemd service, starts the cluster, and writes /etc/rancher/k3s/k3s.yaml. Fast, but unpinned — never use in production.

Pinned install (production / IaC)
curl -sfL https://get.k3s.io | \
  INSTALL_K3S_VERSION="v1.30.2+k3s1" \
  INSTALL_K3S_EXEC="--disable=traefik --write-kubeconfig-mode=644" \
  sh -

INSTALL_K3S_VERSION pins the exact release. INSTALL_K3S_EXEC passes flags to the server. --write-kubeconfig-mode=644 makes the kubeconfig readable so you can copy it.

Via Ansible (the way we'll do it)

Use the xanmanning.k3s Galaxy role:

# requirements.yml
roles:
  - name: xanmanning.k3s
    version: v3.4.4

# group_vars/k3s.yml
k3s_release_version: v1.30.2+k3s1

    - traefik
  write-kubeconfig-mode: "0644"
k3s_become: true
# playbooks/k3s.yml
- hosts: k3s_servers

That's the entire k3s install playbook. Add multi-node by setting k3s_control_node: true on the first host and joining agents with the auto-generated token. The role handles tokens, kubeconfig fetch, upgrades.

Topologies
Single-node

One machine running k3s server. Control plane and workloads on the same host. SQLite datastore. Perfect for our sprint scope: one EC2 instance replaces the application + network VMs. ~512 MB control plane overhead.

Single-server + agents

One control plane, multiple worker nodes. Workers join with a token from the server. Still SQLite — no HA, but you can scale workloads. Good for "I want more compute capacity" without HA needs.

HA with embedded etcd

3 (or 5) servers, each running a full k3s server. They form an etcd cluster automatically. Workers join any of them. Production HA with no external dependencies. First server starts with --cluster-init, others join with --server https://<first>:6443.

External datastore

Servers point at an external MySQL/Postgres/etcd. No embedded etcd. Useful when you already have a managed database and want to avoid running etcd yourself. Less common.

Storage on k3s

k3s ships with local-path-provisioner as the default StorageClass. When a PVC is created, it provisions a directory on the node where the requesting pod is scheduled — under /var/lib/rancher/k3s/storage/.

  • Pros: zero config, works on any Linux, fast (local disk).
  • Cons: node-local. If a pod is rescheduled to another node, the data doesn't follow. For single-node clusters this is fine. For multi-node, you need a real distributed storage solution.

Multi-node options:

  • Longhorn — distributed block storage built by Rancher specifically for k3s. Replicates volumes across nodes. The recommended k3s upgrade path.
  • NFS — install nfs-subdir-external-provisioner chart, point at an NFS server.
  • CSI for AWS EBS / GCE PD / Azure Disk — if you're on a cloud, install the CSI driver and use cloud volumes.

For our project: single-node EC2 + local-path is fine for sprint 2. Longhorn is the natural upgrade if we add nodes.

Networking on k3s

Default CNI is Flannel with VXLAN backend. Pod CIDR 10.42.0.0/16, service CIDR 10.43.0.0/16. No NetworkPolicy enforcement.

Default ingress is Traefik (which we disable to use Kong). Default LoadBalancer implementation is ServiceLB (klipper-lb), which uses host ports on every node — meaning type: LoadBalancer Just Works without a cloud LB.

If you want NetworkPolicies, swap Flannel for Calico:

--flannel-backend=none --disable-network-policy
# then kubectl apply Calico manifests
Upgrading k3s

Two clean ways:

  • Re-run the install script with a new version pin. The script handles binary swap and service restart.
  • Use the system-upgrade-controller — a k3s-specific operator that drains and upgrades nodes one at a time based on a CRD. Production-grade.

Always read the release notes for breaking changes (especially for K8s minor version bumps like 1.29 → 1.30).

When NOT to use k3s
  • You need to learn K8s internals at the control-plane level — k3s hides them. Use kubeadm.
  • You need a feature k3s explicitly removed (some legacy in-tree storage drivers, alpha features).
  • You're standardizing on a managed cloud K8s — just use EKS/GKE/AKS, don't reinvent.
  • You need very large clusters (hundreds of nodes) — vanilla K8s is more battle-tested at that scale.
Where k3s fits in our stack

The migration shape:

  • Terraform provisions: VPC, security groups, an EC2 instance for the k3s host, RDS, S3/CloudFront. Same as today, just fewer EC2s.
  • Ansible bootstraps the EC2: common role, docker (no longer needed!), then a new k3s role using xanmanning.k3s. Fetches the kubeconfig back to the control node.
  • Helm (driven from Ansible or by hand) installs the workloads: Kong (DB-less, declarative config as ConfigMap), Spring Boot (Deployment + Service), AI namespace (triage, MCP servers, Ollama). Each pinned to a chart version.
  • The monitoring VM stays where it is (per our earlier decision) and reaches into the k3s cluster via kubernetes_sd_configs + an in-cluster OTel Collector DaemonSet.

You delete more roles than you add. Goodbye spring_backend, kong, triage_service, mcp_servers Ansible roles — they all become Helm values files.

k3s configuration via config.yaml

Instead of stuffing flags into INSTALL_K3S_EXEC, you can use a YAML config file that k3s reads on startup. Cleaner for IaC:

# /etc/rancher/k3s/config.yaml
write-kubeconfig-mode: "0644"

  - traefik
  - servicelb            # if you want to use MetalLB instead
disable-network-policy: false
flannel-backend: "vxlan"
node-label:
  - "node-role=worker"
  - "environment=prod"

  - "audit-log-maxage=7"

k3s HA setup — multi-server with embedded etcd

For real HA, run 3 servers (odd number for quorum). The first one initializes the etcd cluster; the others join it.

Manual setup (so you understand what the role does)
# On the first server:
curl -sfL https://get.k3s.io | \
  INSTALL_K3S_VERSION="v1.30.2+k3s1" \
  K3S_TOKEN="my-shared-secret" \
  sh -s - server \
    --cluster-init \
    --tls-san my-loadbalancer.example.com

# Read the join token
cat /var/lib/rancher/k3s/server/node-token

# On the 2nd and 3rd servers:
curl -sfL https://get.k3s.io | \
  INSTALL_K3S_VERSION="v1.30.2+k3s1" \
  K3S_TOKEN="<node-token-from-server-1>" \
  sh -s - server \
    --server https://<server-1-ip>:6443 \
    --tls-san my-loadbalancer.example.com

Backup & recovery

k3s with embedded etcd auto-snapshots etcd to /var/lib/rancher/k3s/server/db/snapshots/ every 12 hours by default. Override:

k3s_server:
  etcd-snapshot-schedule-cron: "0 */4 * * *"     # every 4 hours
  etcd-snapshot-retention: 10                    # keep last 10
  etcd-s3: true                                  # also push to S3
  etcd-s3-bucket: cires-k3s-backups
  etcd-s3-region: eu-west-1

To restore: stop k3s, run k3s server --cluster-reset --cluster-reset-restore-path=<snapshot>, restart. Read the docs carefully — restore is a one-shot operation.

For SQLite (single-node) k3s, "backup" means copying /var/lib/rancher/k3s/server/db/state.db when k3s is stopped. Put it in a cron job.

Monitoring k3s itself

k3s exposes Prometheus-compatible metrics — but unlike kubeadm where each control plane component has its own port, k3s exposes everything from the kubelet's metrics endpoint:

  • https://<node>:10250/metrics — kubelet metrics (cAdvisor included)
  • https://<node>:10250/metrics/cadvisor — container metrics
  • https://<node>:10250/metrics/probes — probe metrics
  • https://<node>:6443/metrics — API server metrics (needs auth)
  • https://<node>:10257/metrics — controller-manager (k3s exposes via --bind-address arg)

For our project: the external Prometheus on the monitoring VM will scrape these endpoints via kubernetes_sd_configs, with bearer token auth from a ServiceAccount in the cluster.

metadata: { name: writer }
spec:
  containers:
  - name: w
    image: busybox
    command: ['sh', '-c', 'date >> /data/log; sleep 3600']
    volumeMounts: [{ name: data, mountPath: /data }]
  volumes:
  - name: data
    persistentVolumeClaim: { claimName: data }
EOF

kubectl exec writer -- cat /data/log
kubectl delete pod writer
# Recreate the same pod, verify the data is still there
kubectl apply -f writer-pod.yaml
kubectl exec writer -- cat /data/log    # previous date is preserved
Exercise 2.4: namespaces and isolation
kubectl create namespace dev
kubectl create namespace staging
# Deploy hello to both
kubectl apply -f hello.yaml -n dev
kubectl apply -f hello.yaml -n staging
# Try to reach across namespaces from inside a pod:
kubectl exec -n dev hello-xxxxx -- wget -qO- hello.staging.svc.cluster.local
# Notice that pod IPs differ but service names are stable
Exercise 2.5: Job and CronJob
kubectl create job hello-once --image=busybox -- echo "hello from a job"
kubectl logs job/hello-once

kubectl create cronjob hello-cron --image=busybox --schedule="*/1 * * * *" -- date
# Wait 2-3 minutes, then:
kubectl get jobs
kubectl get pods --selector=job-name --sort-by=.metadata.creationTimestamp
Exercise 2.6: DaemonSet
cat <<EOF | kubectl apply -f -
apiVersion: apps/v1
kind: DaemonSet
metadata: { name: pinger }
spec:
  selector: { matchLabels: { app: pinger } }
  template:
    metadata: { labels: { app: pinger } }
    spec:
      containers:
      - name: p
        image: busybox
        command: ['sh', '-c', 'while true; do echo "alive on \$NODE_NAME"; sleep 10; done']
        env:
        - name: NODE_NAME
          valueFrom:
            fieldRef: { fieldPath: spec.nodeName }
EOF

kubectl get pods -l app=pinger -o wide
# Expected: one pod per node. Add a node → another pod appears.

Checkpoint: for any new piece of software, you know whether to model it as a Deployment, StatefulSet, DaemonSet, or Job. You've used the downward API (fieldRef) to inject node info into a pod.

Phase 3 — Networking & configuration depth

Goal: you can debug "why can't service A talk to service B" without panicking. This is the phase where K8s stops feeling like magic.

Exercise 3.1: in-cluster DNS resolution
kubectl run -it --rm dnstest --image=nicolaka/netshoot --restart=Never -- bash
# Inside the pod:
nslookup hello                                    # short form (same namespace)
nslookup hello.default                            # ns-qualified
nslookup hello.default.svc.cluster.local          # FQDN
cat /etc/resolv.conf                              # see the search domains
dig +trace hello.default.svc.cluster.local
Exercise 3.2: Service vs pod IPs
# From the netshoot pod:
curl -v http://hello                              # via Service (load balanced)
curl -v http://<some-pod-ip>                      # direct to one pod
# Then on the host:
kubectl get endpoints hello                       # see the actual pod IPs the Service routes to
kubectl get svc hello -o yaml                     # see selector + port mapping
Exercise 3.3: read kube-proxy's iptables rules
# SSH into a node (or k3d's node container):
docker exec -it k3d-learn-server-0 sh
iptables -t nat -L KUBE-SERVICES -n | grep hello
iptables -t nat -L KUBE-SVC-XXXXX -n              # use the chain name from above
# You'll see DNAT rules sending traffic to pod IPs with random load balancing
Exercise 3.4: probes that fail on purpose
# Patch the deployment to set a deliberately broken liveness probe
kubectl patch deploy hello --type=strategic -p '
spec:
  template:
    spec:
      containers:
      - name: web
        livenessProbe:
          httpGet: { path: /this-does-not-exist, port: 80 }
          initialDelaySeconds: 5
          periodSeconds: 5
          failureThreshold: 2'

kubectl get pods -l app=hello -w
# Watch RESTARTS column climb. The app is being killed by kubelet on probe failure.
# Patch it back to a working path to stop the loop.
Exercise 3.5: NetworkPolicy (needs Calico/Cilium)
cat <<EOF | kubectl apply -f -
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all-ingress
  namespace: default
spec:
  podSelector: {}                  # all pods in the namespace
  policyTypes: [Ingress]
  # no ingress rules = deny all
EOF

# Now try to reach hello — it should fail.
kubectl run -it --rm test --image=busybox --restart=Never -- wget -qO- --timeout=3 http://hello

# Allow only pods labeled access=yes:
cat <<EOF | kubectl apply -f -
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-from-labeled
spec:
  podSelector: { matchLabels: { app: hello } }
  policyTypes: [Ingress]
  ingress:
  - from:
    - podSelector: { matchLabels: { access: "yes" } }
EOF

# A pod without the label still fails. Add the label, it succeeds.
kubectl run -it --rm test --image=busybox --labels=access=yes --restart=Never \
  -- wget -qO- http://hello
Exercise 3.6: resource requests and OOMKill
# Create a pod that allocates too much memory
kubectl run hog --image=polinux/stress \
  --requests='memory=100Mi' --limits='memory=100Mi' \
  -- stress --vm 1 --vm-bytes 200M --vm-hang 1

kubectl get pod hog -w
# Expected: OOMKilled, then restart, OOMKilled, restart, → CrashLoopBackOff
kubectl describe pod hog | grep -A3 "Last State"

Checkpoint: when an app can't reach another app, you know to check (in order): DNS resolution → Service exists with correct selector → Endpoints object lists the right pod IPs → pods are Ready → NetworkPolicy doesn't block → iptables rules look sane. You can do all of these from kubectl in under 30 seconds.

Phase 4 — Helm & real software

Goal: you can install third-party software the way the rest of the world does.

  1. helm install the kube-prometheus-stack chart into a monitoring namespace. Browse the resulting pods, services, configmaps with k9s.
  2. Read the chart's values.yaml on Artifact Hub. Override one value (e.g. Grafana's admin password) via -f values.yaml.
  3. Install Loki + OpenTelemetry Collector via their Helm charts.
  4. Write your own tiny Helm chart with helm create, edit the template, install it.
  5. Practice helm upgrade, helm rollback, helm uninstall.

Checkpoint: you can take any Helm chart from Artifact Hub, read its values, customize, and install it.

Phase 5 — RBAC, security, & CRDs

Goal: you understand the security model and the extensibility model.

  1. Create a ServiceAccount, a Role granting it get/list/watch on pods, a RoleBinding. Verify with kubectl auth can-i.
  2. Pull the ServiceAccount token from inside a pod (cat /var/run/secrets/.../token) and use it with curl against the API server.
  3. Install cert-manager via Helm. Notice it installs CRDs (Certificate, Issuer). kubectl get crds to see them.
  4. Read the source of one operator (cert-manager or the Prometheus Operator) — even just skimming the controller's reconcile function teaches a lot.

Checkpoint: you can grant a workload the minimum permissions it needs and you understand what an operator is doing under the hood.

Phase 6 — Production patterns

Goal: ready to operate a real cluster.

  1. Set up a HorizontalPodAutoscaler against a real load test.
  2. Practice draining a node (kubectl drain), watching pods reschedule, then uncordon.
  3. Adopt GitOps: set up ArgoCD or Flux pointing at a git repo of manifests. Make a change in git, watch the cluster sync.
  4. Configure resource quotas and LimitRanges per namespace.
  5. Read about and experiment with PodDisruptionBudgets, taints/tolerations, affinity/anti-affinity.
  6. Read the K8s audit log for one day on a real cluster. You'll learn an enormous amount about what's actually happening.
Optional Phase 7 — Internals (kubeadm walkthrough)

Goal: stop being mystified by what the control plane is doing.

  1. On a throwaway VM, install vanilla K8s with kubeadm init.
  2. Look at every static pod manifest under /etc/kubernetes/manifests/ — read the flags.
  3. Use etcdctl (with the certs in /etc/kubernetes/pki/etcd/) to read keys directly from etcd.
  4. Move /etc/kubernetes/manifests/kube-scheduler.yaml away. Watch new pods stay Pending. Move it back. You now feel what the scheduler does.
  5. Tear it all down with kubeadm reset.

Skip this if time-pressed; come back to it when you want depth.
