Step 3 — Generate Ansible Inventory from Terraform
3

Run generate-inventory.sh to auto-populate IPs

The Ansible inventory (production.yml) has been parameterized — all host IPs, the RDS endpoint, and the CloudFront domain are now variables resolved from group_vars/all.yml. Instead of editing files manually, run the generator script from the Terraform repo. It reads terraform output and writes the values directly into your Ansible config.

# From the provisioning-monitoring-infra repo, after terraform apply:
./generate-inventory.sh ../monitoring-project

# The script does three things:
#   1. Reads terraform output (IPs, RDS endpoint, CloudFront domain)
#   2. Writes them into monitoring-project/inventory/group_vars/all.yml
#   3. Updates production.yml ansible_host values
How it works: The inventory now uses variables like monitoring_vm_ip, application_vm_ip, network_vm_ip, and ai_vm_ip defined in group_vars/all.yml. Cross-group endpoints (loki_push_endpoint, jaeger_endpoint, prometheus_host) are derived from these variables automatically. No more hardcoded IPs.

The generated variables in group_vars/all.yml look like this:

Ansible Migration -- What Changed and Why
6

Alertmanager Removal

WHY it was removed: In Sprint 1, Alertmanager received firing alerts from Prometheus and routed them to Gmail. In Sprint 2, we need alerts to reach the AI Triage Service as a webhook so the LLM can perform root-cause analysis. Grafana Alerting can do everything Alertmanager did (SMTP, webhooks, grouping, silencing) plus it can evaluate queries against both Prometheus AND Loki from a single UI. Keeping Alertmanager alongside Grafana Alerting would create duplicate notifications and confusing ownership of alert routing. Removing it eliminates that confusion.

What was deleted:

  • roles/alertmanager/ -- entire directory removed (tasks, templates, defaults, handlers)
  • alertmanager service from monitoring-compose.yml.j2 (container, ports :9093, volumes, health check)
  • alertmanager scrape target from monitoring.yml group_vars (prometheus_targets list)
  • alertmanager Grafana datasource from datasources.yml.j2
  • alertmanager status panel from the unified-overview dashboard JSON

What was modified:

  • monitoring.yml playbook -- removed - alertmanager from the roles list and removed the "Wait for Alertmanager" health check task
  • prometheus.yml.j2 -- removed the alerting: block (which pointed to alertmanager:9093) and removed rule_files: (which loaded alert-rules.yml)
  • monitoring-compose.yml.j2 -- removed the alert-rules.yml volume mount from the prometheus service

prometheus.yml.j2 -- BEFORE (Sprint 1):

  scrape_interval: {{ prometheus_scrape_interval }}

Alert Rules Migration to Grafana

WHY: Grafana Alerting can evaluate queries against both Prometheus and Loki from a single rule engine. This means we can have Loki-based log alerts (like ingestion rate dropping) alongside Prometheus metric alerts, all managed in one place with one notification pipeline. The Triage Service webhook receives a unified JSON payload regardless of which datasource triggered the alert.

OLD format (Prometheus alert-rules.yml.j2 -- Sprint 1):

# This file was mounted into Prometheus as /etc/prometheus/alert-rules.yml
groups:
  - name: application_alerts
    rules:
      - alert: HighP95Latency
        expr: histogram_quantile(0.95, sum(rate(http_server_requests_seconds_bucket{job="app-spring-actuator"}[5m])) by (le)) * 1000 > 1000
        for: 2m
        labels:
          severity: warning
        annotations:
          summary: "P95 request latency exceeds 1s"

NEW format (Grafana alertrules.yml.j2 -- Sprint 2):

apiVersion: 1

groups:
  - orgId: 1
    name: tracing_alerts
    folder: Provisioned Alerts
    interval: 60s         # Grafana evaluates every 60s
    rules:
      - uid: high-p95-latency
        title: HighP95Latency
        condition: C       # Which refId decides firing
        data:
          # Step A: Run the PromQL query (range query over last 10 min)
          - refId: A
            datasourceUid: prometheus
            model:
              expr: 'histogram_quantile(0.95, sum(rate(http_server_requests_seconds_bucket{job="app-spring-actuator"}[5m])) by (le)) * 1000'
              instant: false
              range: true
              intervalMs: 15000
            relativeTimeRange:
              from: 600        # 600 seconds = 10 minutes ago
              to: 0            # now
          # Step B: Reduce the time series to a single number (last value)
          - refId: B
            datasourceUid: __expr__   # Special: server-side expression
            model:
              type: reduce
              expression: A             # Takes output of refId A
              reducer: last             # Gets the last data point
          # Step C: Compare against threshold (fires if > 1000ms)
          - refId: C
            datasourceUid: __expr__
            model:
              type: threshold
              expression: B             # Takes output of refId B
              conditions:
                - evaluator:
                    type: gt              # Greater than
                    params: [1000]       # 1000ms threshold
        for: 2m             # Must fire continuously for 2 min
        labels:
          severity: warning
        annotations:
          summary: "P95 request latency exceeds 1s"
          description: "The 95th percentile latency has been above 1000ms for 2 minutes."
The 3-step pipeline pattern (A -> B -> C): Every Grafana alert rule follows the same structure: refId A runs the datasource query (PromQL/LogQL), refId B reduces the time series to a single value (last, mean, max, etc.), and refId C applies a threshold comparison (gt, lt). The condition: C field tells Grafana that refId C is the one that determines firing state.

MySQL to RDS Migration

WHY: Running MySQL in a Docker container on the application VM is fine for a lab, but AWS best practice is to use a managed database service (RDS). RDS handles automated backups, patching, failover, and monitoring. This also simplifies the application VM -- it only runs Spring Boot, OTel Collector, node-exporter, and cAdvisor.

What was removed from application-compose.yml.j2:

  • mysql service -- the entire container block (image, env vars, volumes, health check)
  • mysql-exporter service -- no longer needed since RDS provides its own CloudWatch metrics
  • mysql_data volume -- no local data volume needed
  • depends_on: mysql -- Spring Boot no longer waits for a local MySQL container

Kong Frontend Route to CloudFront

WHY: In Sprint 1, the React frontend was served by the same Spring Boot container (bundled in the JAR). Kong proxied / to Spring Boot. In Sprint 2, the React build artifacts are deployed to S3 and served via CloudFront CDN. This is AWS best practice for static frontends -- lower latency, HTTPS by default, no load on the backend for static files.

kong.yml.j2 -- BEFORE (Sprint 1):

services:
  - name: app-frontend
    url: http://{{ app_backend_host }}:{{ app_frontend_port }}  # Pointed to Spring Boot
    plugins:
      - name: opentelemetry   # OTel tracing on frontend route
        config:
          traces_endpoint: "http://otel-collector:4318/v1/traces"
    routes:
      - name: frontend-route
        paths: [/]

kong.yml.j2 -- AFTER (Sprint 2):

services:
  # Backend service — unchanged, still has OTel plugin for tracing
  - name: app-backend
    url: http://{{ app_backend_host }}:{{ app_frontend_port }}
    plugins:
      - name: opentelemetry
        config:
          traces_endpoint: "http://otel-collector:4318/v1/traces"
          header_type: w3c
          resource_attributes:
            service.name: "kong-gateway"
          propagation:
            extract: [w3c, b3]
            inject: [w3c]
            default_format: w3c
    routes:
      - name: api-route
        paths: [/api]
        strip_path: false
      - name: actuator-route
        paths: [/actuator]
        strip_path: false

  # Frontend service — NOW proxies to CloudFront (S3 static site)
  - name: app-frontend
    url: https://{{ cloudfront_domain }}   # CloudFront URL from Terraform output
    routes:                                 # NO OTel plugin — static assets don't need tracing
      - name: frontend-route
        paths: [/]
        strip_path: false
Two key changes: (1) The frontend service URL changed from http://backend:80 to https://CLOUDFRONT_DOMAIN. (2) The OTel plugin was removed from the frontend service -- static HTML/CSS/JS files served from S3 don't produce meaningful trace data. The OTel plugin remains on the backend service where /api and /actuator routes generate actual distributed traces.

# Step 1: Add NVIDIA Container Toolkit GPG key to trusted keyring
# The 'creates' argument makes this idempotent — skips if key file exists
- name: Add NVIDIA Container Toolkit GPG key
  shell: |
    curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | \
      gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
  args:
    creates: /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg
  when: nvidia_runtime_enabled | default(false)

# Step 2: Add the NVIDIA apt repository (signed with the GPG key from step 1)
- name: Add NVIDIA Container Toolkit repository
  shell: |
    curl -s -L https://nvidia.github.io/libnvidia-container/stable/deb/nvidia-container-toolkit.list | \
      sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
      tee /etc/apt/sources.list.d/nvidia-container-toolkit.list
  args:
    creates: /etc/apt/sources.list.d/nvidia-container-toolkit.list
  when: nvidia_runtime_enabled | default(false)

# Step 3: Install the toolkit package (apt-get install nvidia-container-toolkit)
- name: Install NVIDIA Container Toolkit
  apt:
    name: nvidia-container-toolkit
    state: present
    update_cache: yes
  when: nvidia_runtime_enabled | default(false)

# Step 4: Tell Docker to use the nvidia runtime (modifies /etc/docker/daemon.json)

# ---- VPC: The virtual network that contains everything ----
resource "aws_vpc" "main" {
  cidr_block           = var.vpc_cidr     # 10.0.0.0/16 = 65,536 IPs
  enable_dns_support   = true            # Allow DNS resolution inside VPC
  enable_dns_hostnames = true            # EC2 instances get DNS hostnames

  tags = { Name = "${var.project_name}-vpc" }
}

# ---- PUBLIC SUBNET: Where the NAT Gateway lives ----
# Only the NAT Gateway sits here. No EC2 instances are in the public subnet.
resource "aws_subnet" "public" {
  vpc_id                  = aws_vpc.main.id
  cidr_block              = var.public_subnet_cidr   # 10.0.1.0/24
  availability_zone       = var.availability_zone    # us-east-1a
  map_public_ip_on_launch = true

  tags = { Name = "${var.project_name}-public" }
}

# ---- PRIVATE SUBNET A: Where all EC2 instances and RDS live ----
resource "aws_subnet" "private" {
  vpc_id            = aws_vpc.main.id
  cidr_block        = var.private_subnet_cidr    # 10.0.2.0/24
  availability_zone = var.availability_zone     # us-east-1a

  tags = { Name = "${var.project_name}-private" }
}

# ---- PRIVATE SUBNET B: Required by RDS subnet group (AWS mandates 2 AZs) ----
# AWS RDS requires a DB Subnet Group with subnets in at least 2 Availability Zones.
# This subnet exists ONLY to satisfy that requirement. No instances are placed here.
resource "aws_subnet" "private_b" {
  vpc_id            = aws_vpc.main.id
  cidr_block        = "10.0.3.0/24"              # Hardcoded (not in variables)
  availability_zone = "${var.aws_region}b"       # us-east-1b (different AZ)

  tags = { Name = "${var.project_name}-private-b" }
}

Since SGs are referenced by EC2 instances, deleting them first would fail. create_before_destroy creates the replacement SG first, updates references, then deletes the old one.
All egress is open: Every SG has egress 0.0.0.0/0 all ports. This is intentional for a demo -- instances need to reach Docker Hub, apt repos, Gmail SMTP, etc. In production, you would restrict egress to specific endpoints.

Design decision — env_prefix="": No prefix is used, so environment variables map 1:1 to field names (e.g., OLLAMA_MODEL=llama3.1:8b). This keeps the ai-compose.yml.j2 clean and readable. The case_sensitive=False flag means both OLLAMA_URL and ollama_url work.

Key setting groups and what they control:

  • Ollama settings: Docker service name ollama:11434, model name llama3.1:8b, and a 300-second timeout (LLM inference on 8B params can be slow without GPU).
  • MCP URLs: Each MCP bridge server is a separate Docker service. The triage service calls them via HTTP. Defaults use Docker DNS names.
  • Loki direct URL: Used only by the Drain3 background ingestion loop, which polls Loki directly (not via the Loki MCP) to continuously build log templates.
  • SMTP: Gmail App Password authentication. If smtp_user is empty, the notifier silently skips email sends (graceful degradation during development).
  • Context timeouts: Each MCP source gets 8 seconds. If Prometheus is down, Loki and Jaeger results are still used. The pipeline continues with partial context.
  • Pipeline timeout (300s): Hard safety limit. If the entire pipeline (context + Drain3 + LLM + email) exceeds 5 minutes, a raw alert email is sent instead.
  • Drain3 anomaly threshold (5): A log template is flagged as [ANOMALY] if fewer than 5 matching lines exist in the Drain3 corpus. This distinguishes "never seen before" from "normal baseline."

Step 5 — Alert Deduplication (dedup.py)

app/dedup.py — DedupManager with TTL Windows

Grafana Alerting re-sends firing alerts every evaluation interval (typically 1 minute). Without deduplication, the same alert would trigger 5+ LLM investigations in a 5-minute window. The DedupManager uses a keyed TTL dictionary to suppress duplicate alerts.

@dataclass
class DedupEntry:
    first_seen: float
    count: int = 1
    last_status: str = "firing"


class DedupManager:
    def __init__(self, window_seconds: int = 300):
        self.window = window_seconds
        self._entries: dict[tuple[str, str], DedupEntry] = {}
        self._lock = asyncio.Lock()

    async def check(self, alertname: str, instance: str, status: str) -> bool:
        """Returns True if the alert is a duplicate that should be skipped."""
        key = (alertname, instance)
        now = time.monotonic()

        async with self._lock:
            self._cleanup(now)

            if key not in self._entries:
                # First time seeing this alert — process it
                self._entries[key] = DedupEntry(first_seen=now, last_status=status)
                return False

            entry = self._entries[key]

            # If alert resolved and re-fired, reset window and re-process
            if status == "firing" and entry.last_status == "resolved":
                entry.first_seen = now
                entry.count = 1
                entry.last_status = status
                return False

            # Track resolved status but don't process
            if status == "resolved":
                entry.last_status = "resolved"
                return True

            # Duplicate within window — skip
            entry.count += 1
            return True

    def _cleanup(self, now: float):
        # Remove entries older than the dedup window
        expired = [k for k, e in self._entries.items() if now - e.first_seen > self.window]
        for k in expired:
            del self._entries[k]
Why asyncio.Lock? FastAPI processes multiple webhook POSTs concurrently (async). Without a lock, two simultaneous webhooks for the same alert could both pass the "not in entries" check and both trigger LLM investigations. The lock serializes only the dedup check — the actual pipeline processing runs concurrently after the check.

Three important behaviors:

  • Key = (alertname, instance): The same alert on different instances is NOT deduplicated. A HighLatency alert on app-vm:8080 and network-vm:8000 are treated as separate incidents.
  • Resolved + re-fired: If an alert resolves within the 5-minute window and then fires again, the dedup window resets and the new firing is processed. This prevents missing a new incident that happens to share the same alert name.
  • Periodic cleanup: Every call to check() first removes expired entries. This is O(n) but n is small (typically under 20 active alerts). No background timer thread needed.

app/drain_analyzer.py — Log Anomaly Detection via Template Mining

Drain3 is a streaming log template miner. It clusters log lines into templates (e.g., "GET /api/users <NUM> <IP> 200") and tracks how many times each template has been seen. The DrainAnalyzer uses this to flag anomalous log lines — lines that match new or rarely-seen templates.

Core method — analyze():

class DrainAnalyzer:
    def __init__(self):
        os.makedirs(settings.drain3_state_dir, exist_ok=True)
        persistence = FilePersistence(
            os.path.join(settings.drain3_state_dir, "drain3_state.bin")
        )
        self._miner = TemplateMiner(persistence, config_filename="drain3.ini")

    def analyze(self, log_line: str) -> AnalyzeResult:
        result = self._miner.add_log_message(log_line)
        cluster = result.get("cluster_id")
        change_type = result.get("change_type", "none")
        is_new = change_type in ("cluster_created", "cluster_template_changed")

        # Count how many lines match this cluster
        match_count = 0
        if cluster is not None:
            for c in self._miner.drain.clusters:
                if c.cluster_id == cluster:
                    match_count = c.size
                    break

        # Anomalous = new pattern OR fewer than threshold matches
        if is_new or match_count < settings.drain3_anomaly_threshold:
            self._total_anomalies += 1

        return AnalyzeResult(cluster_id=cluster, template=template,
                            is_new_pattern=is_new, match_count=match_count)

This helps the LLM focus on genuinely unusual log patterns.

            alerts_deduplicated.inc()
            return

        # Step 2: Wrap everything in a timeout
        try:
            await asyncio.wait_for(
                self._investigate_and_act(alert, source, pipeline_start),
                timeout=settings.pipeline_timeout,   # 300 seconds
            )
        except asyncio.TimeoutError:
            # Safety fallback: send raw alert email
            pipeline_timeouts.inc()
            await self.notifier.send_timeout_alert(alert)
            emails_sent.labels(type="timeout").inc()
            # Record timeout in RCA history
            record = RCARecord(
                alert_source=source, alert_name=alert.alertname,
                triage_decision="timeout_passthrough",
                action_taken="emailed_raw", ...)
            await self.store.save_decision(record)

The inner _investigate_and_act() runs the full investigation pipeline:

    async def _investigate_and_act(self, alert, source, pipeline_start):
        # Step 3: Gather context from all three pillars (parallel)
        ctx = await self.context.gather(alert.alertname, alert.service, alert.startsAt)

        # Step 4: Annotate logs with Drain3
        if ctx.logs:
            annotated_logs, anomaly_summary = self.drain.annotate_lines(ctx.logs)
            ctx.annotated_logs = annotated_logs
            ctx.anomaly_summary = anomaly_summary
            # Update Prometheus metrics
            drain3_lines_processed.inc(len(ctx.logs))
            drain3_anomalies.inc(sum(1 for l in annotated_logs if l.startswith("[ANOMALY]")))
            drain3_clusters.set(self.drain.get_stats()["total_clusters"])

        # Step 5: Check RCA history for prior occurrences
        history = await self.store.get_alert_frequency(alert.alertname)
        history_context = ""
        if history["count"] > 0:
            history_context = f"This alert has fired {history['count']} time(s) ..."

        # Step 6: Call LLM for investigation
        decision, llm_ms = await self.llm.investigate(
            alert, ctx, anomaly_summary, history_context)
        llm_duration.observe(llm_ms / 1000)

        elapsed_ms = int((time.monotonic() - pipeline_start) * 1000)
        pipeline_duration.observe(elapsed_ms / 1000)

        # Step 7: Build RCA record
        record = RCARecord(
            alert_source=source, alert_name=alert.alertname,
            llm_verdict=decision.decision.value.lower(),
            rca_report=decision.rca,
            action_taken="emailed" if decision.decision == Decision.ESCALATE else "suppressed",
            investigation_duration_ms=elapsed_ms, ...)

        # Step 8: Act on decision
        if decision.decision == Decision.ESCALATE:
            await self.notifier.send_escalation(alert, decision, record, history["count"])
            emails_sent.labels(type="escalation").inc()
            alerts_processed.labels(decision="escalate").inc()
        else:
            alerts_processed.labels(decision="dismiss").inc()

        # Step 9: Save to RCA history
        await self.store.save_decision(record)
The timeout fallback (Step 2) is the most important safety mechanism. If Ollama is unresponsive or the T4 GPU is saturated, the entire pipeline could hang. asyncio.wait_for(..., timeout=300) ensures that after 5 minutes, the on-call engineer still receives an email with the raw alert data. The timeout is recorded in SQLite as triage_decision="timeout_passthrough" and the triage_pipeline_timeouts_total Prometheus counter is incremented — you can alert on this metric to detect when Ollama needs more resources.

Why 202 Accepted + BackgroundTasks? LLM inference can take 30-120 seconds. If the webhook handler waited for the full pipeline to complete, Grafana's webhook sender would time out and retry, creating duplicate processing. By returning 202 Accepted immediately and processing in the background, the webhook is acknowledged in milliseconds. The BackgroundTasks mechanism runs the pipeline in the same event loop (no separate worker needed) after the response is sent.

Loki Retention for RCA Logs

Keep RCA logs longer than regular application logs

RCA outputs are valuable for post-incident review. Consider a longer retention period for the llm-triage job. Add a stream selector rule to the Loki config:

# In roles/loki/templates/loki.yml.j2 — per-stream retention (Loki 2.9+)
compactor:
  working_directory: /loki/compactor
  retention_enabled: true

limits_config:
  retention_period: 720h   # 30 days default
  per_stream_rate_limit: 3MB

# Per-stream retention override for RCA logs — 90 days
ruler:
  storage:
    type: local
