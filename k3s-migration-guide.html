What Changed and Why

Sprint 2 was extended to April 23 with a new epic: migrate the monitored infrastructure (Spring Boot, Kong, AI RCA pipeline, Ollama) from per-VM docker-compose to a single-node k3s cluster on EC2.

Key decision: The monitoring stack (Prometheus, Grafana, Loki, Jaeger) stays on its own VM. Only the monitored targets move to Kubernetes. This keeps complexity manageable and avoids circular dependencies (you don't want your monitoring to go down because your cluster has issues).

Why k3s (not full K8s, not Docker Swarm, not Nomad)

What is k3s?

k3s is a lightweight, certified Kubernetes distribution built by Rancher (now SUSE). It packages the entire Kubernetes control plane into a single ~70MB binary. It's fully conformant — anything that works on vanilla K8s works on k3s.

Why we chose it

Criterionk3skubeadm (full K8s)Docker Swarm
Install complexitySingle binary, 30 secondsMultiple components, 30+ minBuilt into Docker
Resource overhead~512MB RAM for control plane~2GB RAM minimum~200MB
Certified K8sYes (CNCF conformant)YesNo (different API)
Helm supportYes (via helm binary)YesNo
GPU supportYes (NVIDIA device plugin)YesYes (--gpus flag)
Ansible automationYes (xanmanning.k3s role)Yes (kubespray)Yes (manual)
Production pathSame K8s API → EKS/GKE laterAlready full K8sDead end (Docker deprecated Swarm)
Bottom line: k3s gives us real Kubernetes with minimal overhead. When we move to CIRES private cloud, we can upgrade to a multi-node k3s cluster or move to kubeadm/EKS — the Helm charts work everywhere.

What k3s Includes (built-in)

  • containerd — container runtime (replaces Docker)
  • CoreDNS — cluster DNS
  • Flannel — CNI networking
  • local-path-provisioner — PersistentVolume storage
  • Traefik — ingress controller (we disable this because we use Kong)
  • ServiceLB — basic load balancer
  • Metrics Server — resource metrics for HPA

What k3s Does NOT Include

  • Docker (uses containerd directly — docker CLI doesn't work on the k3s node)
  • Dashboard (install separately if needed)
  • Monitoring (that's our monitoring VM)
  • NVIDIA GPU support (added via device plugin in E3-06)

Key Commands

# kubectl is aliased by k3s
sudo k3s kubectl get nodes

# Or use the fetched kubeconfig
export KUBECONFIG=~/.kube/config

How Data Flows (K8s Version)

  1. Metrics: Prometheus on monitoring VM discovers K8s pods via kubernetes_sd_configs. Pods annotated with prometheus.io/scrape: "true" are automatically scraped.
  2. Logs: OTel Collector DaemonSet reads /var/log/pods/, enriches with K8s metadata (pod, namespace, container), forwards to Loki on monitoring VM.
  3. Traces: OTel Java Agent (Spring Boot) and Kong OTel plugin send OTLP to in-cluster OTel Collector DaemonSet, which forwards to monitoring VM OTel Collector → Jaeger.
  4. Alerts: Grafana evaluates rules (now using K8s pod selectors), fires webhook to triage service (reached via NodePort or internal service DNS).

Terraform Changes (SCRUM-63 Rescope)

The original Terraform provisioned 4 separate EC2 instances. The rescoped version provisions:

Ansible & k3s Bootstrap (E3-01)

How k3s is Installed

We use the xanmanning.k3s Ansible role (v3.4.4) which handles the full k3s lifecycle: install, configure, upgrade, and uninstall.

Step 1: Install the Role

# requirements.yml
roles:
  - name: xanmanning.k3s
    version: "v3.4.4"

# Install
ansible-galaxy install -r requirements.yml

Step 2: Configure Inventory

# inventory/production.yml
all:
  children:
    monitoring:
      hosts:
        monitoring-vm:
          ansible_host: 10.0.2.x  # from Terraform
    k3s_cluster:
      hosts:
        k3s-node:
          ansible_host: 10.0.2.y  # from Terraform
          k3s_control_node: true
    gpu:
      hosts:
        gpu-vm:
          ansible_host: 10.0.2.z  # from Terraform

Step 3: Group Variables

# inventory/group_vars/k3s_cluster.yml
k3s_release_version: "v1.30.6+k3s1"
k3s_server:
  disable:
    - traefik   # We use Kong instead
  write-kubeconfig-mode: "0644"
image_tag: "latest"

Step 4: Run the Playbook

# Bootstrap k3s
ansible-playbook playbooks/k3s.yml

# Verify
kubectl get nodes
# NAME       STATUS   ROLES                  AGE   VERSION
# k3s-node   Ready    control-plane,master   1m    v1.30.6+k3s1

Image Build & Import (E3-02)

Since we have a single-node k3s cluster with 1-2 contributors, we don't need a Docker registry. Instead, we build images with Docker and import them directly into k3s's containerd.

# The flow for each first-party image:
docker build -t cires/triage-service:latest .
docker save cires/triage-service:latest -o /tmp/triage.tar
sudo k3s ctr images import /tmp/triage.tar

# Verify
sudo k3s crictl images | grep cires
Why not a registry? ECR + GitHub Actions adds operational complexity we don't need for a single-node demo. When we go multi-node, we'll add ECR. The upgrade path is documented in the role README.

Helm Charts (E3-03, E3-04, E3-05)

What is Helm?

Helm is the package manager for Kubernetes. A Helm chart is a collection of YAML templates that define Kubernetes resources (Deployments, Services, ConfigMaps, Secrets). You install a chart with helm install and it creates all the resources.

Our Three Charts

ChartLocationTypeInstalls
Spring Bootcharts/spring-boot/Custom (in-house)Deployment, Service, ConfigMap, Secret, ServiceAccount
Kongcharts/kong/Upstream valuesUses kong/kong chart with our values.yaml
AI Stackcharts/ai-stack/Custom (in-house)7 Deployments, 7 Services, PVCs, Secrets, ConfigMap

Installing the Charts

# 1. Spring Boot
helm install spring-boot charts/spring-boot/ \
  -n app --create-namespace \
  --set database.url="jdbc:mysql://RDS_ENDPOINT:3306/app_db" \
  --set database.password="YOUR_DB_PASSWORD"

# 2. Kong (apply ConfigMap first)
kubectl apply -f charts/kong/kong-config.yaml
helm repo add kong https://charts.konghq.com
helm install kong kong/kong \
  --version 2.38.0 \
  -f charts/kong/values-k3s.yaml \
  -n network --create-namespace

# 3. AI Stack
helm install ai-stack charts/ai-stack/ \
  -n ai --create-namespace \
  --set monitoring.prometheusUrl="http://MONITORING_IP:9090" \
  --set monitoring.lokiUrl="http://MONITORING_IP:3100" \
  --set monitoring.jaegerUrl="http://MONITORING_IP:16686" \
  --set smtp.user="your@gmail.com" \
  --set smtp.password="app-password"

Helm Key Concepts

  • Chart.yaml — chart metadata (name, version)
  • values.yaml — default configuration (override with --set or -f)
  • templates/ — Kubernetes YAML with Go template syntax ({{ .Values.xxx }})
  • _helpers.tpl — reusable template functions (name, labels, etc.)
  • helm install — deploy a chart
  • helm upgrade — update a deployed chart
  • helm rollback — revert to a previous version
  • helm uninstall — remove everything the chart created

Debugging Tips

# Preview what Helm will create (dry run)
helm install --dry-run --debug spring-boot charts/spring-boot/

# See what's deployed
helm list -A

# See the actual K8s resources
kubectl get all -n app

Observability Integration (E3-07, E3-08, E3-09)

OTel Collector DaemonSet (E3-07)

A DaemonSet runs one pod per node. Our OTel Collector DaemonSet:

  • Reads container logs from /var/log/pods/
  • Enriches with K8s metadata (pod name, namespace, container) via k8sattributes processor
  • Receives OTLP traces from application pods
  • Forwards logs to Loki on monitoring VM
  • Forwards traces to OTel Collector on monitoring VM → Jaeger
# Apply the OTel Collector manifests
kubectl apply -f manifests/otel-collector/namespace.yaml
kubectl apply -f manifests/otel-collector/rbac.yaml
kubectl apply -f manifests/otel-collector/configmap.yaml
kubectl apply -f manifests/otel-collector/daemonset.yaml

# Verify
kubectl get pods -n observability
kubectl logs -n observability -l app.kubernetes.io/name=otel-collector

Prometheus kubernetes_sd_configs (E3-08)

Instead of manually listing scrape targets, Prometheus discovers K8s pods automatically. Pods opt-in with annotations:

metadata:
  annotations:
    prometheus.io/scrape: "true"
    prometheus.io/port: "8080"
    prometheus.io/path: "/actuator/prometheus"

Setup Steps

  1. Apply RBAC manifests to create the ServiceAccount + ClusterRole in the k3s cluster
  2. Extract the SA token
  3. Place the token file on the monitoring VM at /etc/prometheus/k3s-token
  4. Set k3s_sd_enabled: true in monitoring.yml group_vars
  5. Re-run ansible-playbook playbooks/monitoring.yml
# Apply RBAC
kubectl apply -f manifests/prometheus-rbac/rbac.yaml

# Extract token
kubectl get secret prometheus-scraper-token -n observability \
  -o jsonpath='{.data.token}' | base64 -d > /tmp/k3s-token

# Copy to monitoring VM
scp /tmp/k3s-token deploy@monitoring-vm:/etc/prometheus/k3s-token

Grafana Alert Rule Migration (E3-09)

The 17 alert rules are updated to use K8s-compatible selectors when k3s_sd_enabled is true. The Jinja2 template switches between VM job labels and K8s pod labels automatically.

Old Selector (VM mode)New Selector (K8s mode)
job="app-spring-actuator"app="spring-boot"
job="otel-collector"app="otel-collector"
job="mon-node-exporter"job="mon-node-exporter" (unchanged — monitoring VM stays)
Risk: If any alert rule selector is wrong, the rule silently stops firing. Test each rule individually after migration.

GPU & NVIDIA Setup (E3-06)

Why GPU?

Ollama runs LLM inference (llama3.1:8b). Without GPU, inference takes 30-60 seconds per query. With T4 GPU, it takes 3-5 seconds. The AI RCA pipeline has a 5-minute timeout, so GPU is strongly preferred.

The Stack

  1. NVIDIA host driver — installed on the EC2 host (nvidia_docker Ansible role)
  2. NVIDIA Container Toolkit — configures containerd with nvidia runtime
  3. NVIDIA k8s Device Plugin — DaemonSet that registers nvidia.com/gpu as a schedulable resource
  4. Ollama pod — requests nvidia.com/gpu: 1 in its resource limits

Setup Steps

# 1. Install NVIDIA driver (via Ansible)
ansible-playbook playbooks/k3s.yml --tags nvidia

# 2. Verify on the host
ssh deploy@gpu-vm nvidia-smi

# 3. Deploy device plugin
kubectl apply -f manifests/nvidia-device-plugin/daemonset.yaml

# 4. Test GPU access
kubectl apply -f manifests/nvidia-device-plugin/test-gpu-pod.yaml
kubectl logs gpu-test
# Should show nvidia-smi output with T4 GPU info

# 5. Clean up test
kubectl delete pod gpu-test

End-to-End Validation (E3-10)

The Validation Playbook

ansible-playbook playbooks/integration-k8s.yml

10-Step Test Sequence

  1. Verify k3s node is Ready
  2. Verify all workload pods are Running (Spring Boot, Kong, Triage, Ollama)
  3. Verify monitoring VM services (Prometheus, Grafana, Loki, Jaeger)
  4. Verify Prometheus K8s targets are being scraped
  5. Generate load through Kong NodePort
  6. Verify Spring Boot metrics appear in Prometheus
  7. Verify logs appear in Loki with K8s labels
  8. Verify traces appear in Jaeger
  9. Verify triage service is healthy
  10. Verify Ollama is responding with loaded models
This replaces E1-05 (the original VM-target E2E validation). The K8s validation is the new ground truth.

Troubleshooting

Common Issues

Pod in ImagePullBackOff

# First-party images must be imported into k3s containerd
# They are NOT pulled from Docker Hub
sudo k3s crictl images | grep cires

# If missing, re-run the image import
ansible-playbook playbooks/k3s.yml --tags images

# Ensure imagePullPolicy is IfNotPresent (not Always)
kubectl get deploy -n ai triage-service -o yaml | grep imagePullPolicy

Pod in CrashLoopBackOff

# Check pod logs
kubectl logs -n ai deploy/triage-service --previous

# Check events
kubectl describe pod -n ai triage-service-xxx

# Common causes:
# - Missing environment variables
# - Database connection refused (RDS security group)
# - Config file not mounted

Prometheus Not Scraping K8s Targets

# Check Prometheus targets page
curl http://monitoring-vm:9090/api/v1/targets | jq '.data.activeTargets[] | select(.labels.job=="kubernetes-pods")'

# Common causes:
# - k3s-token not on monitoring VM
# - Security group blocks :6443 from monitoring VM
# - Pod missing prometheus.io/scrape annotation
# - k3s_sd_enabled not set to true in monitoring.yml

Logs Not Appearing in Loki

# Check OTel Collector DaemonSet logs
kubectl logs -n observability -l app.kubernetes.io/name=otel-collector

# Verify OTel Collector can reach Loki
kubectl exec -n observability otel-collector-xxx -- \
  curl -s http://MONITORING_VM_IP:3100/ready

# Common causes:
# - OTel Collector config has wrong monitoring VM IP
# - Network/security group blocks :3100 from k3s EC2
# - filelog receiver not matching pod log paths

k3s Won't Start

# Check service status
sudo systemctl status k3s

# Check logs
sudo journalctl -u k3s --no-pager -n 100

# Common causes:
# - Port 6443 already in use
# - Insufficient disk space
# - DNS resolution issues
