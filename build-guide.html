Sprint 1 Observability Platform
Full-stack observability for mukundmadhav/react-springboot-mysql deployed across three VMs via Ansible + Docker Compose. All instrumentation is injected externally — OTel Java agent on Spring Boot, Kong OTel plugin at the gateway layer. Metrics, structured logs, and distributed traces are collected, correlated, and visualised in Grafana.
monitoring-vm 192.168.127.10
Prometheus :9090 Alertmanager :9093 Loki :3100 OTel Collector :4317/:4318 Jaeger :16686 Grafana :3000 node_exporter :9100 cAdvisor :8081

Stack at a Glance

📊 Metrics

Prometheus scrapes all VMs. node_exporter for host metrics, cAdvisor for container metrics, MySQL Exporter for DB metrics. Spring actuator scraped via Kong at :8000/actuator/prometheus. All alerts managed by Alertmanager with Gmail SMTP.

📋 Logs

OTel Collector filelog receiver on application-vm and network-vm reads Docker container logs from /var/lib/docker/containers and ships them to Loki on monitoring-vm. Spring Boot logs include trace_id enabling log↔trace correlation in Grafana.

🔍 Traces

Kong OTel plugin sends gateway-level spans to OTel Collector (:4318 HTTP). Spring Boot OTel Java agent sends app spans to OTel Collector (:4317 gRPC). OTel Collector batches and forwards to Jaeger over the internal Docker network. W3C Trace Context + B3 headers.

🔌 Kong Gateway

Declarative dbless config. Routes /api and /actuator to Spring Boot backend, / to React frontend. OTel plugin propagates W3C + B3 trace context. Prometheus plugin exposes Kong metrics. Single proxy entry point at :8000.

📈 Grafana

Three dashboards: unified-overview (RED metrics + logs + traces), tracing-overview (Jaeger explorer, span distribution), otel-collector-health (span drop rate, exporter health). Datasources cross-linked via trace_id.

⚙️ Ansible

15 roles orchestrated by three playbooks (monitoring.yml, application.yml, network.yml) unified under site.yml. Each VM runs one /opt/stack/docker-compose.yml templated by Jinja2. Variables flow from group_vars/ down to roles/*/defaults/main.yml.

ℹ️
Ansible control node: 192.168.127.20 — not managed by Ansible, but added to UFW allowlists on all three VMs. Remote user is deploy with passwordless sudo. SSH key at ~/.ssh/ansible_key (pre-configured in ansible.cfg).
Prerequisites & Deployment
Steps to bring up the full stack from scratch. Run from the Ansible control node (192.168.127.20).
1

Install Ansible and dependencies

Requires Ansible 2.14+ and Python 3.9+ on the control node.

pip install ansible
ansible-galaxy install -r requirements.yml
2

Configure SSH key

Generate or copy the deploy key to ~/.ssh/ansible_key. The deploy user on each VM must have this public key in ~/.ssh/authorized_keys.

ssh-keygen -t ed25519 -f ~/.ssh/ansible_key -N ""
ssh-copy-id -i ~/.ssh/ansible_key.pub deploy@192.168.127.10
ssh-copy-id -i ~/.ssh/ansible_key.pub deploy@192.168.127.30
ssh-copy-id -i ~/.ssh/ansible_key.pub deploy@192.168.127.15
ℹ️
ansible.cfg already sets private_key_file = ~/.ssh/ansible_key and remote_user = deploy — no per-command flags needed.
3

Verify connectivity

ansible all -m ping

All three hosts should return pong. If not, check SSH key, UFW rules, and that the deploy user exists on each VM.

4

Review credentials (important before first deploy)

Weak defaults are set in inventory/group_vars/. Change these before deploying to any non-throwaway environment:

  • grafana_admin_password in monitoring.yml
  • mysql_password and mysql_root_password in application.yml
5

Dry-run (check mode)

ansible-playbook playbooks/site.yml --check

Shows what would change without applying anything. Some tasks (e.g. docker pulls) will still report "changed" because they can't be checked without executing.

6

Full deploy

# Full stack (all three VMs)
ansible-playbook playbooks/site.yml

# Per-VM
ansible-playbook playbooks/monitoring.yml
ansible-playbook playbooks/application.yml
ansible-playbook playbooks/network.yml

# Tagged run — only monitoring-related tasks
ansible-playbook playbooks/site.yml --tags "monitoring"

Each playbook deploys the docker-compose template then starts the stack with docker compose up -d. Total deploy time: ~8–12 minutes for a cold start.

7

Verify services

# Check all containers are running on each VM
ansible monitoring -m shell -a "docker compose -f /opt/stack/docker-compose.yml ps"
ansible application -m shell -a "docker compose -f /opt/stack/docker-compose.yml ps"
ansible network -m shell -a "docker compose -f /opt/stack/docker-compose.yml ps"

Then open in a browser:

  • Grafana: http://192.168.127.10:3000 (admin / your password)
  • Prometheus: http://192.168.127.10:9090
  • Jaeger: http://192.168.127.10:16686
  • Kong proxy: http://192.168.127.15:8000
  • App (via Kong): http://192.168.127.15:8000/
⚠️
Known bootstrap ordering issue: Kong must be running before Prometheus can scrape :8000/actuator/prometheus. If you deploy application-vm before network-vm, Spring metrics will be missing until Kong is up. Deploy in order: monitoring → network → application, or run site.yml which handles this automatically.
Ansible Roles (15)
Click a role to expand. All roles live in roles/. Role-specific defaults are in roles/<name>/defaults/main.yml.
01 common all VMs ▶

Purpose

Bootstraps every VM: installs base packages (curl, git, htop, etc.), sets timezone, configures UFW firewall to allow only required ports + Ansible control node (192.168.127.20).

Key variables

  • common_packages — list in all.yml
  • timezone — default Africa/Casablanca
  • ufw_allowed_ports — per-group port list
ℹ️
UFW rules are additive per group. Ports not explicitly allowed are blocked. The Ansible control node IP (192.168.127.20) is always allowed on all ports.
02 docker all VMs ▶

Purpose

Installs Docker CE and the Compose plugin using the official Docker apt repository. Adds the deploy user to the docker group.

Key variables

  • docker_version — pinned in all.yml
  • docker_compose_version — Compose plugin version
03 node_exporter all VMs ▶

Purpose

Stub role — node_exporter runs as a Docker container managed by the docker-compose template on each VM. This role exists as a hook for any pre-deploy configuration (e.g. textfile collector directories).

Port

:9100 on all three VMs. Prometheus scrapes all three independently.

04 cadvisor all VMs ▶

Purpose

Stub role — cAdvisor runs as a Docker container exposing per-container CPU/memory/network metrics. Prometheus scrapes it at :8081/metrics.

Port

:8081 on all three VMs. Requires /var/run/docker.sock and /sys mounts (defined in compose templates).

05 prometheus monitoring-vm ▶

Purpose

Writes prometheus.yml (scrape config), alert.rules.yml, and the retention/storage config. Prometheus scrapes all targets including Spring actuator via Kong.

Key variables (monitoring.yml)

  • prometheus_scrape_targets — list of target definitions with job_name, targets, metrics_path
  • prometheus_retention — default 15d
  • prometheus_alert_rules — inlined in monitoring.yml
⚠️
Spring actuator target uses metrics_path: /actuator/prometheus and points to Kong (192.168.127.15:8000), not Spring directly. If Kong is down, these metrics stop arriving and will trigger OTel Collector health alerts after 5 minutes.
06 alertmanager monitoring-vm ▶

Purpose

Manages alert routing and notifications. Configured with Gmail SMTP. Two routes: warning severity → email, critical severity → email with shorter repeat interval.

Key variables (monitoring.yml)

  • alertmanager_smtp_from
  • alertmanager_smtp_smarthost
  • alertmanager_smtp_auth_username
  • alertmanager_smtp_auth_password — use Ansible vault
  • alertmanager_email_to
07 loki monitoring-vm ▶

Purpose

Grafana Loki for log aggregation. Configured with TSDB storage backend for efficient label indexing. Receives logs from OTel Collector's Loki exporter.

Key variables

  • loki_retention_period — default 744h (31 days)
  • loki_storage_path — /opt/stack/loki-data
  • loki_push_endpoint in all.yml — used by OTel Collector
08 jaeger monitoring-vm ▶

Purpose

Jaeger v2 with Badger local storage for distributed tracing. Receives traces from OTel Collector over the internal Docker network.

Port remapping

Jaeger v2 uses OTLP ports internally (4317/4318). To avoid conflict with OTel Collector's host ports, Jaeger's host ports are remapped:

  • gRPC: host 4327 → container 4317
  • HTTP: host 4328 → container 4318
  • UI: 16686 (unchanged)
⚠️
OTel Collector → Jaeger uses the Docker internal network, not host ports. The exporter in otel-collector-config.yml targets jaeger:4317 (Docker service name), not localhost:4327.
09 otel-collector monitoring-vm ▶

Purpose

Central telemetry hub. Receives traces via OTLP (gRPC :4317, HTTP :4318). Exposes own metrics at :8888. Exports traces to Jaeger and logs to Loki. Also runs filelog receiver on app-vm and network-vm to ship container logs.

Pipeline

Traces: OTLP receiver → batch processor → OTLP exporter (Jaeger)
Logs: filelog receiver (Docker logs) → Loki exporter
Metrics: Prometheus exporter → scraped by Prometheus

ℹ️
The OTel Collector runs on monitoring-vm for traces. On application-vm and network-vm, the same image runs with only the filelog receiver + Loki exporter pipeline enabled for log shipping.
10 grafana monitoring-vm ▶

Purpose

Grafana with provisioned datasources and three dashboards. Datasource URLs use Docker service names (e.g. http://prometheus:9090) — they never leave the monitoring Docker network.

Files generated

  • provisioning/datasources/datasources.yml
  • provisioning/dashboards/dashboards.yml
  • dashboards/unified-overview.json
  • dashboards/tracing-overview.json
  • dashboards/otel-collector-health.json
⚠️
Grafana datasource URLs use Docker service names, not localhost. They must be on the same Docker network. If you add a datasource pointing to localhost:9090 it will fail from inside the container.
11 kong network-vm ▶

Purpose

API Gateway in dbless (declarative) mode. Routes all traffic to Spring Boot backend. OTel plugin extracts/propagates W3C Trace Context and B3 headers, and sends spans to OTel Collector. Prometheus plugin exposes Kong's own metrics.

Key config (kong.yml)

  • Service: spring-backend → http://192.168.127.30:80
  • Routes: /api, /actuator, /
  • Plugin: opentelemetry → endpoint http://192.168.127.10:4318/v1/traces
  • Plugin: prometheus (built-in)
12 mysql application-vm ▶

Purpose

MySQL 8 with an init SQL script to create the application database and user. MySQL Exporter runs alongside it to expose metrics at :9104/metrics.

Key variables (application.yml)

  • mysql_database
  • mysql_user
  • mysql_password — change from default!
  • mysql_root_password — change from default!
13 spring_backend application-vm ▶

Purpose

Downloads the OTel Java agent JAR, writes a multi-stage Dockerfile that clones mukundmadhav/react-springboot-mysql, builds the Spring Boot app, and starts it with the OTel agent attached. Spring Boot serves both the React frontend (/) and the API (/api) on port 8080.

OTel agent config

  • OTEL_SERVICE_NAME=spring-backend
  • OTEL_EXPORTER_OTLP_ENDPOINT=http://192.168.127.10:4317
  • OTEL_PROPAGATORS=tracecontext,b3multi
  • OTEL_LOGS_EXPORTER=none (logs shipped separately via filelog)
ℹ️
Host port 80 maps to container port 8080. Kong proxies to http://192.168.127.30:80 — not directly to the Docker port. Spring Boot auto-includes trace_id and span_id in log output when the OTel agent is attached.
14 react_frontend application-vm ▶

Purpose

Stub role. The React frontend is bundled into the Spring Boot container during the multi-stage Docker build — there is no separate frontend container. This role exists as a future hook (e.g. if the frontend is ever decoupled).

Note

React is served from / by Spring Boot. Kong routes / → Spring Boot backend at 192.168.127.30:80.

15 otel-collector (filelog, app/network VMs) application-vm network-vm ▶

Purpose

On application-vm and network-vm, the OTel Collector runs with only the filelog receiver + Loki exporter pipeline. It reads Docker container log files from /var/lib/docker/containers/*/*-json.log, extracts trace_id via regex, and ships structured logs to Loki on monitoring-vm.

trace_id extraction

A regex operator in the filelog receiver pipeline extracts trace_id from Spring Boot log lines (MDC format: trace_id=<hex>). This populates a Loki label enabling log↔trace correlation in Grafana.

ℹ️
This is the same otel-collector role as role #9, but deployed on application-vm and network-vm with a different config template that enables only the log shipping pipeline.

📋 Logs Pipeline
Docker container logs → OTel Collector filelog → Loki :3100 → Grafana

OTel Collector runs on application-vm and network-vm with a filelog receiver. It reads from Docker's JSON log files at /var/lib/docker/containers/*/*-json.log.

trace_id extraction

Spring Boot logs include MDC context when the OTel Java agent is attached. The filelog receiver uses a regex operator to extract trace_id from log lines:

operators:
  - type: regex_parser
    regex: 'trace_id=(?P<trace_id>[0-9a-f]{32})'
    parse_to: attributes

This becomes a Loki label, enabling Grafana's "View in Jaeger" link — click a log line, jump to the exact trace.

ℹ️
loki_push_endpoint in all.yml = http://192.168.127.10:3100/loki/api/v1/push. This is the Loki exporter endpoint in the OTel Collector config on app/network VMs.
🔍 Traces Pipeline
Kong OTel plugin → OTel Collector :4318 HTTP
Spring OTel Java agent → OTel Collector :4317 gRPC → Jaeger (internal :4317) → Grafana

Trace propagation

W3C Trace Context (traceparent header) is the primary propagation format. B3 multi-header is also extracted by Kong for compatibility with legacy clients. Both Kong and the Spring OTel agent are configured with tracecontext,b3multi.

OTel Collector config (monitoring-vm)

receivers:
  otlp:
    protocols:
      grpc: {endpoint: "0.0.0.0:4317"}
      http: {endpoint: "0.0.0.0:4318"}
exporters:
  otlp/jaeger:
    endpoint: "jaeger:4317"
    tls: {insecure: true}
pipelines:
  traces:
    receivers: [otlp]
    processors: [batch]
    exporters: [otlp/jaeger]
⚠️
OTel Collector → Jaeger uses the Docker service name jaeger:4317 (internal network), not the host port 4327. This is intentional — it avoids NAT overhead and keeps trace traffic on the internal Docker network.

Prometheus Alert Rules

HighRequestLatency critical
histogram_quantile(0.95, rate(http_server_requests_seconds_bucket[5m])) > 2
Fires when P95 HTTP response time exceeds 2 seconds over a 5-minute window on Spring Boot. Measured via the Spring actuator metrics scraped through Kong. Repeat interval: 10 minutes.
HighSpanDropRate warning
rate(otelcol_processor_dropped_spans_total[5m]) / rate(otelcol_receiver_accepted_spans_total[5m]) > 0.05
Fires when the OTel Collector is dropping more than 5% of incoming spans (e.g. Jaeger is down or the batch queue is full). Indicates trace data loss. Repeat interval: 30 minutes.
OTelCollectorExporterErrors warning
rate(otelcol_exporter_send_failed_spans_total[5m]) > 0
Fires when the OTel Collector exporter is failing to send spans to Jaeger. Any non-zero rate is alertable. Usually indicates Jaeger is down or the internal Docker network is broken. Repeat interval: 15 minutes.

Alertmanager Routing

Alertmanager routes alerts to email (Gmail SMTP). All alert state transitions (firing + resolved) trigger an email. The routing tree:

route:
  receiver: 'email-warning'      # default
  routes:
    - matchers: [severity="critical"]
      receiver: 'email-critical'
      repeat_interval: 10m
    - matchers: [severity="warning"]
      receiver: 'email-warning'
      repeat_interval: 30m

receivers:
  - name: 'email-warning'
    email_configs:
      - to: '{{ alertmanager_email_to }}'
        send_resolved: true

  - name: 'email-critical'
    email_configs:
      - to: '{{ alertmanager_email_to }}'
        send_resolved: true
⚠️
alertmanager_smtp_auth_password should be stored in Ansible Vault, not plaintext in monitoring.yml. Use a Gmail App Password (not your account password) — requires 2FA enabled on the Gmail account.

Health Checks

Check all containers
ansible monitoring -m shell -a \
  "docker compose -f /opt/stack/docker-compose.yml ps"
ansible application -m shell -a \
  "docker compose -f /opt/stack/docker-compose.yml ps"

  "docker compose -f /opt/stack/docker-compose.yml ps"

# On monitoring-vm
docker compose -f /opt/stack/docker-compose.yml \
  logs -f prometheus

# On application-vm
docker compose -f /opt/stack/docker-compose.yml \
  logs -f spring-boot

# On network-vm
docker compose -f /opt/stack/docker-compose.yml \
  logs -f kong
Prometheus targets health

Open http://192.168.127.10:9090/targets — all targets should show UP. Any DOWN targets indicate connectivity or service issues.

# Quick CLI check
curl -s http://192.168.127.10:9090/api/v1/targets \
  | python3 -c "
import json,sys
d=json.load(sys.stdin)
for t in d['data']['activeTargets']:
  print(t['health'], t['scrapeUrl'])"

Restarts & Redeploy

Restart a single service
# SSH to the VM, then:
docker compose -f /opt/stack/docker-compose.yml \
  restart prometheus

# Or via Ansible:
ansible monitoring -m shell -a \
  "docker compose -f /opt/stack/docker-compose.yml \
  restart prometheus"
Redeploy config changes

After changing any group_vars/ or role template, re-run the relevant playbook:

# Monitoring config changed (e.g. alert rules)
ansible-playbook playbooks/monitoring.yml

# Only Prometheus role
ansible-playbook playbooks/monitoring.yml \
  --tags prometheus

# Full redeploy (takes ~10 min)
ansible-playbook playbooks/site.yml
Stop / Start full stack
ansible all -m shell -a \
  "docker compose -f /opt/stack/docker-compose.yml down"

ansible all -m shell -a \
  "docker compose -f /opt/stack/docker-compose.yml up -d"

chmod +x load-test.sh
